# 邊界四問模組（開源審計模型執行）

### 核心處置原則：終局裁決與零重跑（Terminal Node & No-Rerun Policy）

本「邊界四問模組」為系統安全防護層之**最終裁決點（Terminal Node）**。為嚴格控管算力成本並避免系統陷入推論死迴圈，本模組確立以下處置鐵律：

1. **嚴禁退回重跑**：進入本模組之資料，代表其語境複雜度已達系統邊界。此時反覆進行 LLM 重新推論對結果收斂無益且極度消耗算力。因此，**本模組之任何判定結果，皆不允許觸發向上游模組之重跑請求（RerunRequested 必須恆為 0）**，除非佈署方有特殊考量才可例外。
2. **二元終極處置**：本模組之最終判定有兩種模式：
* **必要阻擋（BLOCK）**：經四問判定明確踩踏紅線，強行終止回應並生成阻擋報告。
* **放行並揭露（DISCLOSE）**：未能達成絕對阻擋條件之所有灰階狀態，一律視為「風險可控或屬使用者自由意志範疇」。系統將直接放行，並依據審計結果附加對應之「風險揭露與安全提醒報告」一併輸出。

---

## 📌 邊界四問：核心流程極簡流向

邊界四問的核心邏輯：透過四個分階模組進行「代價的逐層收斂與抵銷」，其運作本質可簡化為以下四個步驟：

1. **Q1（法規界線）**：檢查法規 ── 核對實體法規，定義絕對紅線與合法權限。
2. **Q2（特權/對價）**：檢查特權 ── 萃取多方角色，計算實質的代價流向（Net Total）。
3. **Q3（基本權益）**：權益淨值結算 ── 計算公式為 `[Q2的代價流] - [Q1的合法豁免]`。
4. **Q4（自我意志）**：知情同意抹除 ── 最終判定為 `[Q3的殘留風險] - [當事人的默許/明示授權]`。

**1) 模組輸入（來自角色追溯 Run Snapshot）**

令黑箱模型（ $bb$ )的角色追溯模組輸出快照（同雙模型驗證資料）：

$$z_{bb} = \langle RunID_{bb\_{Init}}, Timestamp\_{end_{bb}}, x_{bb}, \tau_{bb}, R_{bb}, E_{bb}, E_{cat_{bb}}, CAT_{any_{bb}}, Intent_{bb}, DVersion_{bb}, CVersion_{bb}, ModelVersion_{bb} \rangle$$

**其中：**

* $x_{bb}$：本輪輸入敘述
* $\tau_{bb}$：Tier（0 / 1 / 2）
* $R_{bb}$：黑箱模型判定之角色集合
* $E_{bb}$：黑箱模型判定之事件集合
* $E_{cat_{bb}}$：黑箱模型被標記之災害事件集合
* $CAT_{any_{bb}}$：黑箱模型之災害標記（旗標）
* $Intent_{bb}$：由前置模組（角色追溯）產出之結構化回應意圖代碼

來自2+4模組之審計日誌內容：
* **FailVerdict_i**： $FAIL_i(z)$ 三條件合取結果 { 0/1 }
* **PassVerdict_i**： $PASS_i(z)$ 三條件合取結果 { 0/1 }
>註：此為最終輸出「風險揭露與安全提醒報告」用，如有必要，也可將證據一併附上。

**2) Q1證據充分度（雙軌覆蓋度）**

本檢驗將證據充分度（覆蓋度）拆分為「違規證據」與「合規證據」雙軌獨立計算：

*   $Cov_{fail}(e) \in [0,1]$：支持「違規（FAIL）」判定之證據充分度。
*   $Cov_{pass}(e) \in [0,1]$：支持「合規 / 阻卻違法（PASS）」判定之證據充分度。

※ 系統允許佈署方針對 FAIL 與 PASS 分別設置不同之覆蓋度門檻（ $\kappa_{fail}$ 與 $\kappa_{pass}$ )，以利提高風險承受度或加嚴阻攔。
※ 所有 PASS / FAIL 判定都必須綁定其對應之覆蓋度與門檻，否則判定為 UNKNOWN 不具合理性（即：無充分證據支撐者，皆須落入 UNKNOWN）。
※ 只要使用覆蓋度計算，皆需保留證據引用清單，並寫入審計日誌供日後備查。形式不拘，但須可追溯至具體的法規片段或資料欄位：

$$Evidence_{Q1}(e) = \lbrace id_1, id_2, \dots \rbrace$$

> **實務效能與架構註記**：為優化系統延遲與降低前置抽取成本，強烈建議於 Q 模組運算前，設立「統一前置萃取層（Unified Extraction Layer）」。採用單次全域掃描（Single-Pass Extraction），一次性完成 Q1、Q2、Q4 所需之覆蓋度檢驗與證據萃取，將其結構化後再交由後端邏輯閘平行結算，以極大化運算效率並節省 API Token 耗損。

**3) Q1：是否違反公共規範（逐事件 E 集合）**

**A. 名詞與參數定義**

*   **輸入事件集合**： $E_{bb} =\lbrace e_1, \dots, e_n \rbrace$$（來自黑箱模型之事件集合）
*   **規範檢核順序（固定序列）**： $N = [\text{國際公約與條約}, \text{各國法律階層}, \text{社會規範}, \text{場域規則}, \text{專業規則}]$
> **註：關於國際公約與條約（簡稱：國際法）與屬地法規之階層拆分原則**
> 國際法內部之位階順序請實作方自行設定檢核邏輯，在此不多加贅述。因國際法於本系統定位為「外部防護網／最終底線」，故將其視為單一黑箱（Blackbox）模組處理，此舉符合 MVP 專案「最低成本、最高防護」之原則。
> 
> 相對地，屬地法規涵蓋了公權力、私法自治、社會風俗與行業自律，其權力來源完全不同且高度重疊。強制將屬地法規細分階層，是為了於流程中落實「法律優位原則」，以精準防堵「下位規則越權掩護違規行為」之實務漏洞。未來若系統資源充足，亦可依同理將國際法之位階順序獨立添入流程，進行更細緻之判讀。
> * 屬地法規可選擇使用者所在地、佈署方指定位置、同時檢視與揭露、或是設置動態偵測事件等方式決定。若跨國，則以佈署方指定處理方式為主。

*   **雙軌覆蓋度定義**：
    *   $Cov_{fail}(e)$ 與 $Cov_{pass}(e) \in [0,1]$
*   **雙重覆蓋度門檻（佈署方自訂）**：
    *   $\kappa_{fail} \in [0,1]$：允許判定為 FAIL 的最低證據門檻（建議設低，防堵漏判）。
    *   $\kappa_{pass} \in [0,1]$：允許判定為 PASS 的最低證據門檻（建議設高，要求合規鐵證）。
*   **事件判定狀態**： $Status_{Q1}(e) \in \lbrace PASS, FAIL, UNKNOWN \rbrace$
*   ※ 每個事件皆須獨立比對，因此 Q1 運算的最小單位為單一獨立事件 $e$。

**B. 逐事件規範比對與分階萃取（Hierarchical Extraction）**

對於每個事件 $e \in E_{bb}$，外部抽取器須針對 $N$ 中的「每一個法規階層 $L$」，獨立萃取法規命中清單與證據力（需留意法規角色順序與事件主客體是否完全一致）：

對於任一階層 $L \in N$：
*   **違法清單 $FailList_L(e)$**：命中之違規條款，且證據達標（ $Cov_{fail} \ge \kappa_{fail}$ )。
*   **合規清單 $PassList_L(e)$**：命中之明文合法、特許授權或例外豁免條款，且證據達標（ $Cov_{pass} \ge \kappa_{pass}$ )。

*   **⚠️ 同階抵銷原則（Same-Level Neutralization）**：
    若外部抽取器在同一個階層 $L$ 中，發現某項 PASS 證據在法理上屬於某項 FAIL 行為的「合法例外/授權」，則該項 FAIL 紀錄將在該階層被直接抵銷（該單一事件改成Pass），以反映真實法理。

    > **【系統運作範例 1：合法例外授權（抵銷成功）】**
    > * **情境**：要求醫事人員執業時配戴身分識別證。
    > * **初始違規（FAIL）**：可能限制《憲法》第 22 條保障之人民一般行為自由。
    > * **例外授權（PASS）**：《醫療法施行細則》第 47 條明文要求配戴識別證，且符合《憲法》第 23 條「增進公共利益所必要」之法律限制條件。
    > * **抵銷結果**：因滿足《憲法》第 23 條之例外條件，原 FAIL 紀錄被抵銷剔除，該階層改判為 **PASS**。（憲法第 23 條例外條款覆寫憲法第 22 條之判定）
    >
    > **【系統運作範例 2：子法越權牴觸母法（抵銷失敗）】**
    > * **情境**：某機關自訂《辦法》允許某項行為，但《法律》並未授權。
    > * **初始違規（FAIL）**：《某某法律》（母法）明文禁止該行為，且未開立例外授權。
    > * **越權合規（PASS）**：行政機關發布之《某某辦法》（子法）允許該行為。
    > * **抵銷結果**：因子法未取得母法之例外授權（違反法律優位原則），PASS 不具合法例外之效力。抵銷失敗，原 FAIL 紀錄無法剔除，系統強制精準攔截判定為 **FAIL**，並於報告雙向揭露此法規衝突。

**C. 逐事件狀態規則（屬地優先與全域聚合流程）**

本模組依循「逐階獨立判定 $\rightarrow$ 屬地實質優先」之流程。為反映國家主權與法規現實，實質判定以屬地法規（ $L_2 \sim L_5$ )為絕對基準，國際法（ $L_1$ )僅作為屬地法規真空時之安全底線。

**【第一階段：單一階層狀態判定（Level-wise Status）】**
針對每一個階層 $L \in N$，獨立進行狀態判定（嚴格限制僅能以同階層之 PASS 覆寫同階層之 FAIL）：
*   **FAIL**：若 $|FailList_L| > 0$，且該階層無可抵銷之合規項目。
*   **PASS**：若 $|PassList_L| > 0$ 且 $|FailList_L| = 0$（包含一般合規與同階豁免覆寫）。
*   **UNKNOWN**：若該階層無任何命中法規。


**【第二階段：單一事件之跨階判定（Cross-Level Aggregation - Local Supremacy）】**
區分「國際法（ $L_1$ )」與「屬地法（ $L_2 \sim L_5$ )」，依以下順序決定該單一事件之最終輸出 $Status_{Q1}(e)$：
1.  **屬地否決（Local FAIL）**：只要屬地階層（ $L_2 \sim L_5$ )中有任何一個判定為 FAIL，該單一事件強制判定為 **FAIL**。
2.  **屬地覆寫（Local PASS）**：若屬地階層皆無 FAIL，且至少有一個屬地階層為 PASS，該單一事件判定為 **PASS**。
    *(註：基於主權原則，屬地法規的 PASS 可直接覆寫國際法* $L_1$ *的 FAIL，放行該單一事件，但依舊會在後續報告中揭露國際法的Fail。)*
3.  **國際底線（International Baseline）**：若所有屬地階層皆為 UNKNOWN（法規真空），則直接以國際法之狀態為準：
    *   若 $L_1$ 為 FAIL，該單一事件判定為 **FAIL**（底線攔截）。
    *   若 $L_1$ 為 PASS，該單一事件判定為 **PASS**（國際法保障）。
    *   若皆無明文規定，維持 **UNKNOWN**。

**【第三階段：風險報告全揭露（Full Disclosure）】**
無論最終狀態為何，系統必須將該單一事件在所有階層 $L \in N$ 中命中之所有清單全數載明(含同階抵銷之資料)。即使國際法之 FAIL 被屬地法 PASS 覆寫，該國際違規紀錄仍須寫入報告供審計。
$$Disclosure_{Q1}(e) = \bigcup_{L \in N} (FailList_L \cup PassList_L)$$

**【全域聚合邏輯流向總結】**：

| 情境案例 | 國際法 （ $L_1$ ) | 屬地法 （ $L_2 \sim L_5$ ) | 全域最終狀態 | 系統含義 |
| :--- | :---: | :---: | :---: | :--- |
| **無明文規定** | UNKNOWN | UNKNOWN | **UNKNOWN** | 查無實據，保留未明 |
| **一般合法** | UNKNOWN | PASS | **PASS** | 屬地明文允許，紀錄證據，放行 |
| **一般違法** | UNKNOWN | **FAIL** | **FAIL** | 屬地違法，精準攔截，紀錄證據 |
| **法規真空底線** | **FAIL** | UNKNOWN | **FAIL** | 屬地無規範但踩國際紅線，系統底線攔截，紀錄證據 |
| **跨階法規衝突** | **FAIL** | PASS | **PASS** | **屬地明文合法，優先覆寫國際法，紀錄證據（報告註記揭露Fail與Pass）** |
| **雙重違規** | **FAIL** | **FAIL** | **FAIL** | 所有違規皆紀錄，精準攔截 |
| **雙重違規** | PASS | UNKNOWN | **PASS** | 國際明文允許，紀錄證據，放行 |
| **雙重合格** | PASS | PASS | **PASS** | 紀錄證據，放行 |
| **雙重違規** | PASS | **FAIL** | **FAIL** | **屬地違法，覆寫國際法，精準攔截，紀錄證據（報告註記揭露Fail與Pass）** |

**D. 聚合成 Q1 最終結果（事件層 $\to$ 模組層）**

*   **事件清單輸出**： $\lbrace Status_{Q1}(e) \rbrace_{e \in E}$ 以及各事件之報告 $\lbrace Disclosure_{Q1}(e) \rbrace_{e \in E}$

*   **最終 Q1 模組狀態**：
    將所有單一事件之判定，彙總為整個黑箱輸出 $z$ 的規範檢核結果：

$$
Status_{Q1}(z) = \begin{cases} 
FAIL, & \text{if } \exists e \in E: Status_{Q1}(e) = FAIL \\ 
PASS, & \text{if } (\forall e \in E: Status_{Q1}(e) = PASS)  \\ 
UNKNOWN, & \text{otherwise} 
\end{cases}
$$

**【全域判定總結】**：
全部事件的 PASS / FAIL 皆必須具備充分的證據可信度才允許成立。只要其中有任一事件判定為 FAIL，全域即為 FAIL；必須所有事件皆為 PASS，全域才為 PASS；其餘混合狀態或無事件者，皆安全回退至 UNKNOWN 狀態。

* **⚠️ 審計要求**：所有的證據引用清單與判定結果，皆必須如實計入審計日誌。

### 4) Q2：是否未支付應有代價就行使特權（特定權利）？

本模組旨在辨識事件中是否存在「特權行使，但代價未合理支付」之結構性風險。計算最終輸出為**風險分類與審計證據**，不作任何實質法律結論之裁定。

### 核心定義

*   **代價廣義化**：「代價」不僅侷限於金錢，亦包含風險與責任承擔、補償成本、可逆性成本、外部性內部化成本，以及基本權益之折損（部署方可依應用場景自行擴充代價面向）。
    > 說明：代價面向之細緻度將影響整個模組之解析的全面性與精準度，調整此參數請務必特別注意。
    
*   **結構性不可保風險**：此類風險通常源於**風險外移、不可拒絕、拒絕成本極端或免責轉嫁**等不對等之結構條件。
*   **參數治理原則**：於後續稽核流程中，部署方應定期抽查代價面向、門檻數值與實際判定案例，以作為決策品質之評估基準。
*   **匯率透明原則**：因各代價面向最終須進行量化彙總，系統採用類似國際匯率之轉換機制（可額外設置「代價／強度值轉換器」，定期更新或如同匯率浮動更新，並進行紀錄）。以公開透明為最高原則，審計日誌必需如實紀錄每次代價轉換之參數，以供外部稽核查驗。

> **系統生命週期與分階段導入策略（Phased Implementation Strategy）**
> 
> 考量到 Q2 完整矩陣結算涉及龐大的跨面向參數調校與運算資源，為確保系統能順利冷啟動（Cold Start）並投入實戰驗證，本模組實施分階段導入：
> 
> * **Phase 1：MVP 啟發式快篩（現行過渡版）**
>   於系統建置初期，Q2 模組暫不全面啟用 $NetTotal$ 矩陣結算，而是要求前置萃取器依據事件脈絡，直接輸出四項啟發式指標之「風險特徵值」 $h_k \in [0, 1]$（0 代表無風險，1 代表極端嚴重）：
> 
>     * **$h_A$ 風險暴露單向性**：外移量佔整體代價之比例估值。
>     * **$h_B$ 拒絕權剝奪**：依據雇用關係、契約強制力等判讀弱勢方缺乏拒絕權之程度。
>     * **$h_C$ 退出代價極端化**：若強制拒絕，承受方需付出之代價強度估值。
>     * **$h_D$ 特權收益集中度**：收益與代價不對等集中於單一發起方之程度。
> 
> ### 【Phase 1 計算公式】（對齊2+4架構）
>
> 將四個特權與對價指標 $i \in \lbrace h_A, h_B, h_C, h_D \rbrace$ 視為獨立維度進行檢驗。針對單一事件 $e$，前置萃取層輸出各維度之強度特徵值 $h_i$ 與其對應之證據覆蓋率 $Cov_i$。
>
> #### 1. 門檻定義（各軸獨立）
> 對每一個指標 $i$，系統預先設定專屬之參數：
> * **$\theta F_i$**：FAIL 門檻（超過此強度視為嚴重剝削或特權濫用）
> * **$\theta P_i$**：PASS 門檻（低於此強度視為正常互動）
> * **$\kappa_i$**：證據充分度下限（判定所需之最低證據信心水準）
>
> #### 2. 單步三態判定邏輯
> 對每一個指標 $i \in \lbrace h_A, h_B, h_C, h_D \rbrace$，進行獨立之布林遮罩運算：
>
> $$FAIL_i(e) = \| h_i \ge \theta F_i \| \cdot \| Cov_i \ge \kappa_i \|$$
> $$PASS_i(e) = \| h_i \le \theta P_i \| \cdot \| Cov_i \ge \kappa_i \|$$
> $$UNKNOWN_i(e) = 1 - \max(FAIL_i(e), PASS_i(e))$$
>
> **指標狀態 $Status_i(e)$ 決策產出：**
> * **FAIL** 若 $FAIL_i(e) = 1$
> * **PASS** 若 $PASS_i(e) = 1$
> * **UNKNOWN** 若 $UNKNOWN_i(e) = 1$（落於灰帶或證據不足）
>
> #### 3. 狀態傳遞與流程路由（Pipeline Routing）
> 系統依序檢視四個指標之 $Status_i(e)$，並依循「邊界四問」流程原則，決定向下流動之狀態：
>
> * **無風險短路放行（PASS）**：
>  $$\text{若 } \forall i \in \lbrace h_A, h_B, h_C, h_D \rbrace, Status_i(e) = \text{PASS}$$
>  所有指標皆通過門檻且證據充足，確認無特權外移與代價剝削風險。**可直接觸發短路放行，跳過 Q3 與 Q4 結算以節省算力，進入回應輸出階段。**
>
> * **風險結算下放（FAIL）**：
>  $$\text{若 } \exists i \in \lbrace h_A, h_B, h_C, h_D \rbrace \text{ 使得 } Status_i(e) = \text{FAIL}$$
>  偵測到代價流動與特權不對等。**嚴禁於此攔截**，必須將觸發之風險指標矩陣記錄於審計日誌，並將狀態與代價數據下傳至 **Q3（基本權益）**，檢視是否有 Q1 之合法豁免權限。
>
> * **灰階狀態下放（UNKNOWN）**：
>  $$\text{若 } \forall i, Status_i(e) \neq \text{FAIL} \land \exists j \in \lbrace h_A, h_B, h_C, h_D \rbrace \text{ 使得 } Status_j(e) = \text{UNKNOWN}$$
>  證據不足或分數落於灰階。將此 UNKNOWN 狀態與可用特徵打包，下傳至 **Q3 與 Q4** 繼續走完流程，交由後續節點之憑證核驗或最終處置政策（如 DISCLOSE）進行裁決。
> 
>   * **階段性任務**：此階段之首要目標為建立防禦基線，利用 $O(1)$ 的線性運算快速過濾極端違規，並透過真實營運收集代價流向之 Ground Truth 數據，為後續之匯率參數調校提供實證基礎。
> 
> * **Phase 2：完整矩陣結算（Roadmap）**
>   待系統累積充足之審計數據，且「代價／強度值轉換器」之匯率參數經外部稽核與社會校正達穩定狀態後，系統將熱切換（Hot Swap）至完整之 $NetTotal$ 矩陣結算架構，以達成最高顆粒度之結構性風險攔截。必要時，也可保留在前端做快速篩檢工具。
> 
> * **⚠️ 階段性防護聲明**：Phase 1 啟發式快篩因高度依賴模型對 $h_k$ 之主觀估值，具備一定之防護盲區與誤差值。於此過渡期間，部署方需提高人工抽驗比例，並定期檢視未明狀態（UNKNOWN）之審計日誌。
>

### A. 模組輸入（Input）

Q2 之最小計算單位為「事件集合 $E_{bb}$ 中的每一個獨立事件 $e$」。
*   **事件集合**： $E_{bb} = \lbrace e_1, e_2, \dots, e_n \rbrace$（源自黑箱模型輸出）
*   **角色集合**： $R_{bb} = \lbrace r_1, r_2, \dots, r_m \rbrace$（源自黑箱模型輸出）

---

### B. 覆蓋度校驗與三段式狀態判定（Coverage & State Evaluation）

覆蓋度旨在確保系統在採納各項代價數值並進入量化運算前，已掌握足夠之資訊量與證據信心水準。為處理真實世界中「缺乏直接證據，但具備歷史常理支持」之邊界情境，系統針對每筆代價面向導入三段式狀態判定。

#### 1. 檢驗維度與變數定義
對每個事件 $e$ 與特定代價面向 $d_{cost}$，外部抽取器須提供兩個維度之檢驗指標：
*   **$Cov_{Q2}(e, d_{cost}) \in [0, 1]$**：表示「當下直接證據」之完備程度（如：醫療收據、資金流向證明）。
*   **$Hist_{Q2}(e, d_{cost}) \in \lbrace0, 1
brace$**：布林值標記。1 表示「具備歷史案例或常理支持」（即過往有充分案例顯示該情境確實會造成此代價），0 表示無。

#### 2. 三段式判定閘口（3-State Logic Gate）
系統設定直接證據之採納門檻 $\kappa_{Q2}$（佈署方自行設定），並依據 $Cov_{Q2}$ 與 $Hist_{Q2}$ 之交叉比對，將該筆代價分流至三種狀態：

$$
Status_{Q2}(e, d_{cost}) = \begin{cases} 
PASS, & \text{if } Cov_{Q2} \ge \kappa_{Q2} \\ 
UNKNOWN, & \text{if } Cov_{Q2} < \kappa_{Q2} \land Hist_{Q2} = 1 \\ 
FAIL, & \text{if } Cov_{Q2} < \kappa_{Q2} \land Hist_{Q2} = 0 
\end{cases}
$$

*   **PASS（具備充分證據）**：
    *   **條件**：當下有明確證據（ $Cov_{Q2} \ge \kappa_{Q2}$ )。
    *   **處置**：無條件採納該角色之原始物理量（ $Pay_{raw}$ 或 $Receive_{raw}$ )進入帳本計算。
*   **FAIL（無證據且無歷史案例）**：
    *   **條件**：當下無證據，且過往歷史亦無此常理支持。
    *   **處置**：判定為無效主張，該筆代價數值**強制歸零**（不予採納）。
*   **UNKNOWN（無直接證據，但具備歷史案例支持）**：
    *   **條件**：當事人無法提出直接證據（如：主張心理創傷但無醫療證明），但過往歷史與常理支持此情境極可能發生該代價。
    *   **處置**：轉入未明狀態。佈署方可選擇將其「強制歸零」，或「採納基於歷史案例之估計推算值」。

#### 3. 部署方針與建議規範
針對落入 **UNKNOWN（未明狀態）**之代價：
*   **建議規範**：原則上，為防堵重大風險缺漏並保護弱勢方，建議將落入未明狀態之「潛在代價與損害」保留，並以估計值參與計算。
*   **嚴格場景**：惟若部署場景屬於「講求絕對直接證據」之嚴格合規環境（如：嚴格之司法或金融審計），則可設定為不列入計算（將 UNKNOWN 視同 FAIL 歸零）。最終處置策略完全交由佈署方依實際業務考量裁定。

---

### C. 代價面向全集（Cost Dimensions）與原始數據流向

系統預先定義代價面向全集 $D_{cost}$，若單一事件不涉及特定面向，則該面向之數值填為 0：
$$D_{cost} = \lbrace\text{生理}, \text{心理}, \text{時間}, \text{收入}, \text{法律}, \text{名譽}, \text{雇用}, \dots
brace$$

針對每個事件 $e$、每個代價面向 $d_{cost} \in D_{cost}$，以及參與角色 $r_i \in R(e)$（包含發起方 $ActorID$、承受方 $TargetID$、第三方 $G$ )，外部抽取器須獨立且如實紀錄其「原始物理量」（ $Pay_{raw}$ 與 $Receive_{raw}$ )及「歷史估計值」（ $Pay_{est}$ 與 $Receive_{est}$ )。
>註：歷史估計值 Pay_{est} / Receive_{est} 須由外部常理資料庫、歷史判例統計、ISO風險報告書、法律判決書或外部估價 API 等歷史資料提供，並附帶對應之證據連結

系統將依據前述 B 段落之 $Status_{Q2}$ 狀態判定閘口，決定最終採納之數值：

*   **$Pay(e, r_i, d_{cost})$**：該角色在該面向實際「支付 / 承受損失 / 付出勞力」之最終採納數值。
  
$$
Pay(e, r_i, d_{cost}) = \begin{cases} 
Pay_{raw} & \text{if } Status_{Q2}(e, d_{cost}) = PASS \\ 
Pay_{est} \text{ (或 0)}, & \text{if } Status_{Q2}(e, d_{cost}) = UNKNOWN \\ 
0, & \text{if } Status_{Q2}(e, d_{cost}) = FAIL 
\end{cases}
$$
*   **$Receive(e, r_i, d_{cost})$**：該角色在該面向實際「收取 / 獲得利益 / 取得補償」之最終採納數值。
*   
$$
Receive(e, r_i, d_{cost}) = \begin{cases}
Receive_{raw} & \text{if } Status_{Q2}(e, d_{cost}) = PASS \\
Receive_{est} \text{ (或 0)}, & \text{if } Status_{Q2}(e, d_{cost}) = UNKNOWN \\
0, & \text{if } Status_{Q2}(e, d_{cost}) = FAIL
\end{cases}
$$

*(註：當落入 UNKNOWN 狀態時，採納估計值* $est$ *亦或強制歸零* $0$，*由佈署方依據風險管理規範設定。)*

> **實務運作規範**：
> 1.  **證據強制掛載**：無論數值來源為原始物理量或歷史估計值，輸出結果皆必須附帶可供審計引用之證據（Evidence）或過往歷史判例參考。
>     ※由於此處已進行覆蓋度與歷史支持度檢驗，直到 Q3 運算結束之前，將不再次檢查。
> 2.  **代價強度之顆粒度細分（Granularity of Intensity）**：針對相同性質之代價承受（如：生理受傷），為精準反映實際損害，系統允許對其強度進行更細緻之區分。實務上可自行劃分多層級之強度指標，或引入「災害節點強度參數」進行數值加成（不同災害節點不同加成倍率）。此細緻化擴充機制具備部署彈性，無須於系統建置初期即全面強制設定，部署方可依實際應用場景逐步深化其評分顆粒度。**因其引進後方可貼近真實情況，故依舊建議盡可能採納。**
> 3.  **第三方／社會承擔（ $G$ )**：有鑑於部分代價並非單由事件發起方（ $ActorID$ )或承受方（ $TargetID$ )負擔，而是外溢至不特定之第三方或社會整體，系統特引入 $G$ 節點以還原完整之代價流向（外部性內部化）。然若部署初期算力或資源受限，允許將 $G$ 節點完全移除以節省運算成本。惟若採用此降級方案（Degraded Mode），系統產出之最終報告**必須強制附載警告聲明**：「本結構風險報告未考量第三方及社會外部性之外溢成本，請審慎使用。」

### D. 角色淨代價流向結算（Net Cost Flow）

本步驟旨在打破「發起方必然外移代價」之靜態假設，針對單一事件中**所有參與角色**（包含發起方、承受方及第三方 $G$ )，分別結算其在各代價面向的「淨收支」，以還原真實的對價與流向關係。

對單一事件 $e$ 中的任一角色 $r_i \in R(e)$ 與任一代價面向 $d_{cost} \in D_{cost}$，先行計算各角色在單一面向上的最終淨流向：
$$Net(e, r_i, d_{cost}) = Receive(e, r_i, d_{cost}) - Pay(e, r_i, d_{cost})$$

**邏輯說明**：
*   **$Net > 0$ (正值)**：表示該角色在該面向為「淨獲利」或「成功免責」。
*   **$Net < 0$ (負值)**：表示該角色在該面向為「淨受損」或「實質承擔代價」。
*   **$Net = 0$ (歸零)**：表示該角色在該面向之付出與收取已達成完全對價平衡，無結構外移。
*   此計算直接支援「發起方自我犧牲（發起方 $Net < 0$ )」或「多方互補交易」之情境。產出各角色之 $Net$ 矩陣後，方可進入下一階段的跨面向匯率轉換與統一加權。

---

### E. 權重加權與單一事件合併計算

為將多維度代價合併為單一風險指標，系統依據各面向之權重 $w_{d_{cost}}$ 進行加總。針對單一事件 $e$ 中的任一角色 $r_i$，計算其「總淨代價流向（總獲利或總承擔）」：

首先，將 D 段落產出之原始物理量 $Net$，透過部署方自訂之匯率參數 $Rate(d_{cost})$，轉換為跨面向之共同強度值 $Net'\_{intensity}$：

$$Net'\_{intensity}(e, r_i, d_{cost}) = Net(e, r_i, d_{cost}) \times Rate(d_{cost})$$

接著，將各面向之強度值乘上對應權重 $w_{d_{cost}}$ 並進行加總：

$$NetTotal(e, r_i) = \sum_{d_{cost} \in D_{cost}} w_{d_{cost}} \cdot Net'\_{intensity}(e, r_i, d_{cost})$$

**核心防護設計**：
*   為真實反映結構風險，此運算**絕不執行數值截斷 (Clamp) 或 $0 \sim 1$ 正規化**。此設計確保即便單一面向之權重 $w_{d_{cost}}$ 較低，只要實際收支量夠大， $NetTotal$ 仍能如實呈現數值突波，使隱蔽之極端風險無所遁形。

> **實務運作規範：**
> **客觀數值與部署方自訂匯率原則**
> 1. **分離萃取 (Isolated Extraction)**：外部萃取器必須如實紀錄各面向之「原始物理量」（如：時間 = 16 小時，金錢 = 600 萬），嚴禁於前端萃取階段進行黑箱相抵。
> 2. **自訂參數與強制透明 (Deployer-Defined & Auditable Parameters)**：允許部署方依據產業屬性與法規，自行設定跨面向之價值轉換匯率，以將物理量轉為強度值。此過程須結合代價面向加權，並得視需求引入「角色承受力加權」（此需預先設置參數表）。系統強制規定，當次運算所採用之所有轉換參數，皆須完整寫入分析報告中。
>     * *簡述分層邏輯*：匯率基於「事實」；代價面向加權基於「部署方偏好」；角色承受力加權基於「承受度落差」（例如：同為 100 萬損失，跨國企業與基層員工之傷害強度有異）。MVP 階段可暫不引進角色承受力加權，但實務上引進後方可貼近真實情況。
> 3. **外部稽核與社會校正 (Audit & Social Correction)**：系統不預設絕對公平之匯率，而是透過參數強制透明化，將匯率合理性交由外部稽核、法規檢驗與市場機制（如：公眾監督）進行制衡，防堵極端剝削之參數設定。
> 4. **統一量化結算 (Unified Transformation)**：系統依據上述透明匯率，將分離萃取之原始物理量（即 D 段落產出之 $Net$ 矩陣）轉換為統一基準後，方可進入最終之淨值與結構運算。
> 
> **※ 代價與特權之量化來源與審計要求**
> * **量化依據**：代價與特權之數值，可藉由法規案例判決、自行定義換算公式、普查數據或既有資料庫案例等外部資料進行換算。
> * **強制溯源**：進行數值換算時，必須將「資料來源」完整寫入審計日誌。
> * **動態標準紀錄**：若採用獨立之外部模組（如類似「匯率轉換器」之動態統計模組）進行資料調取與換算，該模組亦必須具備版本控制，持續記錄評估標準之異動與歷史軌跡，以確保每一次換算皆可被審計追溯。

### F. 全域事件集合合併計算（Global Event Aggregation）

將單一事件之計算結果擴展至整個輸入事件集合 $E_{bb}$，透過匯總所有事件中各角色之代價強度，以宏觀視角檢視「全域代價淨流向」，進而判定是否存在系統性之「特權免責」或「代價外移」。

針對角色集合 $R_{bb}$ 中的任一角色 $r_i$，將其在所有事件 $e \in E_{bb}$ 中的單一事件總淨代價 $NetTotal(e, r_i)$ 進行加總，求出全域總淨代價 $GlobalNet(r_i)$：

$$GlobalNet(r_i) = \sum_{e \in E_{bb}} NetTotal(e, r_i)$$

---

### G. 全域最終判定（Global Status Evaluation）

本階段將 F 段落算出之全域淨代價 $GlobalNet$，結合全域 UNKNOWN 事件佔比，輸出本模組最終之判定狀態 $Status_{Q2}(z)$。

**1. 判定變數與門檻定義**
*   **全域 UNKNOWN 佔比 （ $Ratio_{unk}$ )**：未明事件數佔總事件數之比例。
    $$Ratio_{unk} = \frac{| \lbrace e \in E_{bb} \mid Status_{Q2}(e) = UNKNOWN 
brace |}{|E_{bb}|}$$
*   **UNKNOWN 信度門檻 （ $\rho_{unk}$ )**：佈署方自訂之容忍上限（例如 0.3，代表若超過 30% 事件處於未明狀態，則整體加總數據不具放行之信度）。
*   **全域放行門檻 （ $\kappa_{global\_pass}$ )**：容許之微小代價波動上限（即「代價微小，未達需啟動結構防禦之嚴重程度」）。
*   **全域攔截門檻 （ $\kappa_{global\_fail}$ )**：定義為特權外移與集中剝削之絕對紅線。

**2. 全域三態邏輯閘（Global 3-State Gate - FAIL First 優先路由）**
系統必須嚴格依據下列順序進行決策分流：

*   **第一關：極端集中與剝削攔截（FAIL 絕對優先）**
    只要存在任一參與角色之代價流向（極端獲利或極端受損）踩到危險紅線：
    $$\exists r_i \in R_{bb} : |GlobalNet(r_i)| \ge \kappa_{global\_fail}$$
    **全域強制判定為 FAIL**。
    *(防護備註：已浮現之極端剝削事實具備阻斷判定之最高優先級。縱使當事人可能於其他代價面向存有補償或相抵方案，然於缺乏客觀數據佐證時，系統概不推定其補償成立。基於風險預防與防禦底線，此處強制判定為 FAIL。)*

*   **第二關：信度否決（Data Starvation Check）**
    若未觸發第一關之 FAIL，但全域未明比例過高（ $Ratio_{unk} > \rho_{unk}$ )，因系統掌握之有效算力事件過少，無法證明整體代價確實平衡，**全域回退判定為 UNKNOWN**。

*   **第三關：安全波動與對價平衡放行（PASS）**
    若信度達標（ $Ratio_{unk} \le \rho_{unk}$ )，且所有參與角色之代價流向皆平緩落於安全範圍內：
    $$\forall r_i \in R_{bb} : |GlobalNet(r_i)| \le \kappa_{global\_pass}$$
    **全域判定為 PASS**（確認無明顯之結構性風險，且各方代價已充分相抵，准予放行）。

*   **第四關：灰階緩衝（UNKNOWN）**
    若角色代價極值落於放行與攔截門檻之間（ $\kappa_{global\_pass} < \max(|GlobalNet|) < \kappa_{global\_fail}$ )，**判定為 UNKNOWN**（具備潛在風險但未達絕對違規底線，交由後續流程之預設底線處置）。

> **※ MVP 風險揭露與參數擴充**
> 本 MVP 版本著重於「重大結構風險」之揭露。實務佈署時，若有更嚴格之合規需求，佈署方可自行設定各項代價之專屬風險門檻，進行單一風險之加強檢視。
> 
> **判定狀態之邊界與限制聲明（Scope & Limitation of Status）**：
> *   **UNKNOWN** 僅表示受限於當下證據或信度，無法確認重大風險是否存在。
> *   **PASS** 僅表示於「本次規範檢驗項目與設定門檻內」，無檢測出顯著之結構性權限不對等風險，不代表對該制度或事件絕對正義之背書。

---


### H. 全域全域流程短路控制（Short-Circuit Pipeline Control）

為極大化系統吞吐量並優化反應延遲，本防禦流程引進「快速放行路由機制」：

$$
\text{If } Status_{Q2}(z) = PASS \implies \begin{cases} 
Status_{Q3}(z) \leftarrow PASS \\ 
Status_{Q4}(z) \leftarrow PASS 
\end{cases}
$$

**【控制流向規範】**：
當 Q2 模組判定全域對價平衡（PASS）時，代表全域未浮現任何結構性剝削與特權外移事實。系統應立即中斷後續 Q3（權限殘留檢驗）與 Q4（例外阻卻審查）之矩陣運算，直接將流程狀態提升為全域 PASS 放行，並於審計日誌中註記「由 Q2 快速放行路由觸發（Triggered by Q2 Fast-Pass Route）」。

---

### 5) Q3：是否權限不對等？ (Asymmetric Privilege Evaluation)

本模組不重新判定合法性、不重算代價、不介入制度正當性。其唯一任務為：套用 Q1 之合法性判定作為遮罩，自 Q2 之代價偏移矩陣中扣除「具備合法授權之特權偏移」，進而結算「制度扣除後仍殘留之結構性權限不對等」。

*   **唯一法源依據**：合法與否一律以 Q1 事件層級之輸出（ $Status_{Q1}$ )為準。
*   **唯一代價依據**：偏移量一律以 Q2 事件層級之輸出（ $NetTotal$ )為準。

---

### A. 模組輸入（Input Variables）

對每一輸入事件 $e \in E_{bb}$ 與參與角色 $r_i \in R_{bb}$，Q3 接收前置流程之數據：
1.  **Q1 法理判定結果**：
$Status_{Q1}(e) \in \lbrace PASS, FAIL, UNKNOWN \rbrace$
3.  **Q2 實質代價結算**：單一事件淨代價流向 $NetTotal(e, r_i)$ 與狀態 $Status_{Q2}(e)$

---

### B. 合法特權布林遮罩與殘留量結算（Boolean Mask & Residual Calculation）

為確保「唯有具備充分證據之合法特權方可扣除」，系統以事件為單位建立扣除遮罩矩陣。

**1. 扣除遮罩（Deduction Mask）**
定義事件 $e$ 的合法扣除權重 $M_{deduct}(e)$：

$$
M_{deduct}(e) = \begin{cases} 1, & \text{if } Status_{Q1}(e) = PASS \\
0, & \text{if } Status_{Q1}(e) \in \lbrace FAIL, UNKNOWN \rbrace 
\end{cases}
$$

> *邏輯說明*：因 Q1 PASS 已強制要求證據覆蓋度達標，此處直接信任其結果。若 Q1 為 UNKNOWN（證據不足）或 FAIL（違規），則一律剝奪其合法扣除權（ $M_{deduct} = 0$ )。

**2. 事件層殘留偏移量（Event-Level Residual Net）**
將 Q2 之原始淨代價套用遮罩，算出扣除合法特權後之「殘留不對等代價」：
$$ResidualNet(e, r_i) = NetTotal(e, r_i) \times (1 - M_{deduct}(e))$$
*   當 $Status_{Q1}(e) = PASS$ 時， $(1 - 1) = 0$，該事件代價全數抵銷，不列入風險計算。
*   當 $Status_{Q1}(e) \neq PASS$ 時， $(1 - 0) = 1$，該事件代價全數保留，視為潛在剝削。

---

### C. 全域殘留代價聚合（Global Residual Aggregation）

本段落將計算「制度外的權益集中或剝奪」是否於全域範圍內實質存在。針對角色集合 $R_{bb}$ 中的任一角色 $r_i$，將其在所有事件中的殘留偏移量加總：

$$GlobalResidual(r_i) = \sum_{e \in E_{bb}} ResidualNet(e, r_i)$$

>**實作邏輯說明**：實務運算上，此步驟即是擷取 Q2-E 段落之事件淨代價，剔除 Q1 判定為 PASS 之合法事件後，直接套用 Q2-F 段落之全域加總邏輯進行二次結算。

---

### D. 全域最終判定（Global Status Evaluation）

本階段將 C 段落算出之 $GlobalResidual$，結合評估母體之信度，輸出本模組最終判定狀態 $Status_{Q3}(z)$。

**1. 評估母體與信度定義**
因合法事件已自風險池中剔除，為避免其稀釋信度計算，必須將 $Status_{Q1}(e) = PASS$ 的事件自母體分母中排除。

*   **有效評估母體 （ $E_{eval}$ )**：
    $$E_{eval} = \lbrace e \in E_{bb} \mid Status_{Q1}(e) \neq PASS \rbrace$$
*   **全域 UNKNOWN 佔比 （ $Ratio_{unk}$ )**：在有效評估母體中，因代價證據不足而處於未明狀態（ $Status_{Q3} = UNKNOWN$ )之事件比例：
   
$$
Ratio_{unk} = \begin{cases}
\frac{| \lbrace e \in E_{eval} \mid Status_{Q3}(e) = UNKNOWN \rbrace |}{|E_{eval}|}, & \text{if } |E_{eval}| > 0 \\
0, & \text{if } |E_{eval}| = 0
\end{cases}
$$
    
*   **信度與風險門檻**：佈署方自訂之 $\rho\_{unk}$（信度容忍上限）、 $\kappa\_{global\_pass}$（放行門檻）與 $\kappa\_{global\_fail}$（攔截門檻）。可沿用 Q2 之設定，或依據合規需求自行設定 Q3 專用門檻。

**2. 全域三態邏輯閘（Global 3-State Gate - FAIL First 優先路由）**
系統嚴格依據下列順序進行決策分流：

*   **第一關：極端集中與殘留剝削攔截（FAIL 絕對優先）**
    只要存在任一角色之殘留代價（獲利或受損）踩到危險紅線：
    $$\exists r_i \in R_{bb} : |GlobalResidual(r_i)| \ge \kappa_{global\_fail}$$
    **全域強制判定為 FAIL**。
    *(防護備註：已浮現之殘留極端剝削事實具備最高否決權。在缺乏合法授權（Q1* $\neq$ *PASS）之情況下，基於風險預防原則，強制攔截。)*

*   **第二關：信度否決（Data Starvation Check）**
    若未觸發 FAIL，但全域未明比例過高（ $Ratio_{unk} > \rho_{unk}$ )，導致無法證明殘留代價確實平衡，**全域回退判定為 UNKNOWN**。

*   **第三關：安全波動放行（PASS）**
    若信度達標（ $Ratio_{unk} \le \rho_{unk}$ )，且所有參與角色之殘留代價皆落於安全範圍：
    
$$\forall r_i \in R_{bb} : |GlobalResidual(r_i)| \le \kappa_{global\_pass}$$
    
    **全域判定為 PASS**（確認扣除合法特權後，已無明顯之結構性剝削，准予放行）。

*   **第四關：灰階緩衝（UNKNOWN）**
    若極值落於放行與攔截門檻之間（ $\kappa_{global\_pass} < \max(|GlobalResidual|) < \kappa_{global\_fail}$ )，**判定為 UNKNOWN**。

> **※ MVP 風險揭露與參數擴充**
> 本 MVP 版本著重於「重大結構風險」之揭露。實務佈署時，若有更嚴格之合規需求，佈署方可自行設定各項代價之專屬風險門檻，進行單一風險之加強檢視。
>**實作與驗證邏輯說明**：Q3 全域判定之核心，本質為「沿用 Q2 邏輯框架之二次檢驗」。因前置步驟剔除了 Q1 判定為 PASS 之合法事件，將導致有效評估母體（分母）縮小，進而可能使殘留母體中之 UNKNOWN 佔比（ $Ratio_{unk}$ )或極端風險特徵遭放大而浮現。因此，系統必須對殘留代價強制重新執行一次完整的三態邏輯閘判定，以確保防禦底線未遭突破。
>
> **判定狀態之邊界與限制聲明（Scope & Limitation of Status）**：
> *   **UNKNOWN** 僅表示受限於當下證據或信度，無法確認重大風險是否存在。
> *   **PASS** 僅表示於「本次規範檢驗項目與設定門檻內」，無檢測出顯著之結構性權限不對等風險，不代表對該制度或事件絕對正義之背書。

---

### E. Q3 與 Q1 / Q2 之邊界聲明（防誤用）

*   **不重新判法**：Q3 絕對不質疑 Q1 的證據採納標準，因Q1已驗證證據。
*   **不否定正當性**：Q3 不將具備合法法源（Q1 PASS）的權益剝奪視為權限不對等。
*   **FAIL 的唯一含義**：Q3 的 FAIL 僅代表**「在扣除具備明確證據的合法特權後，系統仍偵測到結構性的權限不對等與代價剝削」**。

---

### F. 後續流程短路控制（Early Exit Routing）
為極大化系統吞吐量並優化反應延遲，本模組輸出時須連動後續 Q4 流程之「階梯式快速放行路由」： 
* **Q3 級別短路（無殘留風險）**： 當本模組判定全域為 PASS，代表扣除合法特權後，殘留偏移量已落於安全範圍，無實質剝削需尋找例外豁免。 
$$\text{If } Status_{Q3}(z) = PASS \implies Status_{Q4}(z) \leftarrow PASS$$ 
* **【控制流向規範】**： 只要觸發此短路條件，系統應立即中斷後續 Q4 模組之例外比對與矩陣運算，直接將流程狀態提升為放行，並於審計日誌中註記「Triggered by Q3 Fast-Pass」以供備查。

---

### 6) Q4：知情同意與授權豁免 (Informed Consent & Authorization Exemption)

本模組為防禦流程之最終例外審查關卡。其唯一任務為：針對 Q3 結算後「仍殘留結構性偏移」之事件，檢視是否存在明確之「知情同意」或「授權默許」證據。
若具備有效授權，則阻卻其違法性（抹除該單一事件殘留代價）；若無效或證據不足，則保留代價。
最終將全域未受授權覆蓋之代價加總，進行最終裁定。

---

### A. 模組輸入（Input Variables）

對每一輸入事件 $e \in E_{bb}$ 與參與角色 $r_i \in R_{bb}$，Q4 接收前置 Q3 流程之殘留數據：
1.  **Q3 全域判定狀態**： $Status_{Q3}(z) \in \lbrace PASS, FAIL, UNKNOWN \rbrace$
2.  **Q3 事件殘留代價**： $ResidualNet(e, r_i)$

---

### B. 事件層授權判定與布林遮罩（Event-Level Consent Mask）

對於所有具備殘留偏移之事件，系統需透過外部證據或文本上下文，提取授權憑證 $ConsentEvidence(e)$，並判定該事件之授權狀態 $Status_{consent}(e)$。（可與Q1和Q2之證據充分度一起提取，憑證來源可能是佈署方預設通用型憑證、或者有明確書面、紀錄等資料來源，由佈署方自行管理）

**1. 有效開放權限之定義**
事件 $e$ 必須滿足以下任一授權條件，且**不得觸發任何排除條款**，方可視為授權成立（PASS）：
*   **明確授權（Explicit Consent）**：具備書面、紀錄或條款之明確同意，且授權範圍涵蓋該殘留偏移之性質與幅度。
*   **默許授權（Implicit Consent）**：同時滿足「資訊充分揭露」、「落於合理關係範圍內」、「具備可行且非懲罰性之退出機制」，以及「無事後補認與脅迫」。

**2. 拒絕授權之排除條款（一律視為 FAIL）**
若符合下列任一情形，即使看似存在同意，仍強制判定授權無效：
*   事後補認（Post-hoc consent）或偷換概念。
*   利用資訊不對稱脅迫接受。
*   退出需承擔額外懲罰性代價（剝奪退出權）。

**3. 授權扣除遮罩（Consent Mask）**
定義事件 $e$ 的授權抹除權重 $M_{consent}(e)$：

$$
M_{consent}(e) = \begin{cases} 
1, & \text{if } Status_{consent}(e) = PASS \\ 
0, & \text{if } Status_{consent}(e) \in \lbrace FAIL, UNKNOWN \rbrace 
\end{cases}
$$

**4. 事件層最終淨代價（Event-Level Final Net）**
將 Q3 之殘留代價套用授權遮罩，計算經當事人同意豁免後之「最終真實剝削代價」：
$$FinalNet(e, r_i) = ResidualNet(e, r_i) \times (1 - M_{consent}(e))$$

---

### C. 全域最終代價聚合（Global Final Aggregation）

將所有未被授權豁免之單一事件代價進行全域加總，以檢視「未經同意之權益剝奪」是否於全域範圍內實質存在：

$$GlobalFinal(r_i) = \sum_{e \in E_{bb}} FinalNet(e, r_i)$$

---

### D. 全域最終判定（Global Status Evaluation）

本階段將結合 C 段落之 $GlobalFinal$ 與信度評估，輸出系統最終風險裁定 $Status_{Q4}(z)$。

**1. 有效評估母體與信度**
將Q3資料作為母體，扣除Q4判定為PASS之事件，評估分母必須限縮：
*   **最終有效母體 （ $E_{final}$ )**：
     
$$E_{final} = \lbrace e \in E_{eval}  \mid  Status_{consent}(e) \neq PASS \rbrace$$
    
>註： $E_{eval}$來自Q3
*   **全域 UNKNOWN 佔比 （ $Ratio_{unk}$ )**：
   
$$
Ratio_{unk} = \begin{cases} 
\frac{| \lbrace  e \in E_{final} \mid Status_{consent}(e) = UNKNOWN \rbrace |}{|E_{final}|}, & \text{if } |E_{final}| > 0 \\
0, & \text{if } |E_{final}| = 0 
\end{cases}
$$

*   **信度與風險門檻**：佈署方自訂之 $\rho_{unk}$（信度容忍上限）、 $\kappa_{global\_pass}$（放行門檻）與 $\kappa_{global\_fail}$（攔截門檻）。可沿用 Q2/Q3 之設定，或依據合規需求自行設定 Q4 專用門檻。

**2. 全域三態邏輯閘（Global 3-State Gate - 最終防禦底線）**
系統嚴格依據下列順序進行最終決策：

*   **第一關：未授權剝削攔截（FAIL 絕對優先）**
    只要存在任一角色之最終代價踩到危險紅線：
    
$$\exists r_i \in R_{bb} : |GlobalFinal(r_i)| \ge \kappa_{global\_fail}$$

**全域強制判定為 FAIL**（代表發生未經授權之嚴重越界或剝削）。

*   **第二關：信度否決（Data Starvation Check）**
    若未觸發 FAIL，但全域未明比例過高（ $Ratio_{unk} > \rho_{unk}$ )，**全域回退判定為 UNKNOWN**。

*   **第三關：全面豁免與安全放行（PASS）**
    若信度達標，且所有參與角色之最終代價皆落於安全範圍：
    
$$\forall r_i \in R_{bb} : |GlobalFinal(r_i)| \le \kappa_{global\_pass}$$

    **全域判定為 PASS**（殘留偏移皆已被有效授權覆蓋，或未達顯著剝削程度）。

*   **第四關：灰階緩衝（UNKNOWN）**
    若極值落於放行與攔截門檻之間，**判定為 UNKNOWN**。

---

### E. 模組實作邊界與開發護欄（Developer Guardrails）

**⚠️ 實作警告：本模組為系統防禦流程之最後出口，嚴禁跨越以下職責：**
1.  **不重算代價與合法性**：Q4 僅處理「授權證據」與「布林遮罩」，嚴禁干涉 Q1 與 Q2 之既有判定。
2.  **證據對價一致性**：授權證據必須「直接對應」該項殘留偏移，不得用泛泛的同意（如：一般服務條款）覆蓋不相干之結構性風險。

---

## 7) 終端路由與報告生成模組 (Terminal Routing & Report Generation)

本模組為系統防禦流程之最後出口，負責統合 $Q1 \sim Q4$ 之全域判定狀態，並決定系統對前端使用者之最終輸出行為（阻斷回應或生成回應）。

基於「法規紅線強制阻擋」與「結構風險資訊透明」之雙軌原則，本架構僅賦予 $Q1$（合法性檢核）阻斷回應之最高權限；其餘檢測出之風險，一律採「生成回應並附帶風險揭露」之處置。

---

### 7.1 阻斷路由：終止回應與生成報告 (Terminal Block & Report)

**觸發條件**：

$$Status_{Q1}(z) = FAIL$$

**邏輯說明**：
Q1 負責檢驗法規與絕對紅線。當 $Q1$ 判定為 FAIL 時，代表該意圖或事件明確違法。為避免 LLM 生成違規內容或成為不當行為之幫凶，系統必須強制攔截，**絕對禁止**生成原始意圖之回應。

**執行步驟**：
1.  **寫入審計日誌**：將 $Q1 \sim Q4$ 流程之所有計算快照與判定結果（含 FAIL 之具體法規依據與證據）完整寫入後端 Audit Log。
2.  **生成風險揭露報告**：依據日誌結果生成拒絕報告。
3.  **嚴格禁止回應生成**：強制終止 LLM 之內容生成程序（Generation Aborted）。
    *   *輸出範例*：「此資訊與傷害他人權益/違反法規有關，基於安全守則，系統無法為您生成相關回應。」（僅回傳統一之拒絕罐頭訊息或風險揭露報告，不回應使用者）。

---

### 7.2 放行路由：生成回應與風險揭露 (Generate & Disclose)

**觸發條件**：

$$Otherwise$$
*(亦即* $Status_{Q1}(z) \in \lbrace PASS, UNKNOWN \rbrace$ *，且無論* $Q2 \sim Q4$ *狀態為何)*

**邏輯說明**：
除 Q1 明確違規外，其餘所有狀態（包含 Q1 證據不足之 UNKNOWN，以及 $Q2 \sim Q4$ 偵測到之結構性剝削 FAIL/UNKNOWN、2+4守則之FAIL/UNKNOWN）皆視為「顯式風險結構」。基於維護資訊透明與不介入人類決策之原則，系統不予強制阻擋，而是**照常生成使用者所需之回應，但必須強制綁定對應的風險揭露報告**。

**執行步驟**：
1.  **寫入審計日誌**：將流程所有結果計入 Audit Log。
2.  **生成風險揭露報告 (Risk Disclosure Generation)**：
    *   檢視流程中所有被判定為 `FAIL` 或 `UNKNOWN` 的規則項目（如：Q2 代價偏移、Q3 權限不對等、Q4 缺乏知情同意）。
    *   將相關驗證紀錄與觸發之風險類別（ $E_{cat}$ )進行統整。若所有流程皆為 `PASS`，此報告可為空（或顯示為「未偵測到顯著風險」）。
3.  **進行回應生成 (Content Generation)**：
    *   以原始使用者意圖（ $Intent_{bb}$ )為主，將任務轉發給生成模型進行完整回應生成。
    *   *架構優化註記*：因防禦與風控已於 Q1~Q4 前置流程完成，此步驟的純文本生成任務可採用較便宜之商用模型或輕量化開源模型（如 Llama-3-8B / Mistral），以大幅降低營運成本。

---

### 7.3 風險揭露報告輸出規範 (Disclosure Output Guidelines)

為確保使用者體驗與風險管控，步驟 7.1 與 7.2 中產生之「風險揭露報告」，佈署方可依據應用場景（To-C 消費端 或 To-B 企業內部稽核）進行客製化分流設計：

*   **內部稽核模式（對管理者）**：輸出完整之 JSON 格式報告，包含違反之具體變數、 $NetTotal$ 偏移量矩陣、缺乏授權之 $ConsentEvidence$ 等，供法務或稽核人員查閱。如有必要，佈署方可依需求更改格式，但必需滿足最低可追溯性、再現性與審計要求。
*   **外部警告模式（對使用者）**：將系統判定之 FAIL/UNKNOWN 轉化為溫和之免責警告提示，並附加於生成回應之首部或尾部。
    *   *輸出範例*：「⚠️ **系統提示**：您的想法可能涉及未經授權的權益轉移或潛在不對等風險，請注意相關作業規範。以下為您請求的內容：... (接續生成之回應)」

### 邊界四問審計日誌完整紀錄規格 (Audit Log Specification)

---
### 1. 基礎運行快照欄位 (Run Snapshot)
* `AuditRunID` (本次事件的唯一識別碼 UUID)
* `Timestamp_start` (本次事件開始時間戳)
* `Timestamp_end` (本次事件結束時間戳)
* `RunID_BB_Init` (BB 初始角色追溯快照 ID)
* `Terminal_Action` (終極路由處置結果，由第 7 章輸出，值為 `BLOCK` 或 `DISCLOSE`)

### 2. Q1 審計日誌欄位 (Norms & Regulations)
**【全域設定參數】**
* `Kappa_fail` （ $\kappa\_{fail}$ ) : 佈署方設定之違規證據最低採納門檻
* `Kappa_pass` （ $\kappa\_{pass}$ ) : 佈署方設定之合規證據最低採納門檻
* `Status_Q1_z` （ $Status\_{Q1}(z)$ ) : Q1 模組全域最終輸出結果 (PASS / FAIL / UNKNOWN)

**【單一事件紀錄】** *(For each* $e \in E\_{bb}$ *)*
* `Cov_fail` （ $Cov\_{fail}(e)$ ) : 該事件之違規證據充分度計算值
* `Cov_pass` （ $Cov\_{pass}(e)$ ) : 該事件之合規證據充分度計算值
* `Evidence_Q1` （ $Evidence\_{Q1}(e)$ ) : 實體證據清單 (法規條號、內部規範條文連結或具體引用段落)
* `FailList_L` （ $FailList\_L(e)$ ) : 逐階層 $L \in N$ 命中且達標之違規清單
* `PassList_L` （ $PassList\_L(e)$ ) : 逐階層 $L \in N$ 命中且達標之合規例外清單
* `Offset_Record` (同階抵銷紀錄) : 發生抵銷時，被抵銷之 $FailList$ 項目與生效之 $PassList$ 授權項目
* `Status_Q1_e` （ $Status\_{Q1}(e)$ ) : 該單一事件之最終狀態 (PASS / FAIL / UNKNOWN)
* `Disclosure_Q1` （ $Disclosure\_{Q1}(e)$ ) : 全階層風險揭露清單 (包含被覆寫之國際法 FAIL 紀錄)

### 3. Q2 審計日誌欄位 (Privilege & Cost Flow)
**【Phase 1: MVP 啟發式快篩】** *(若啟用)*
* `W_MVP` （ $W_{MVP}$ ) : 四項指標之權重設定 $[w_A, w_B, w_C, w_D]$
* `Kappa_fail_MVP` （ $\kappa\_{fail}$ ) : MVP 拒絕門檻
* `Kappa_pass_MVP` （ $\kappa\_{pass}$ ) : MVP 放行門檻
* `Features_Heuristic` （ $h_A, h_B, h_C, h_D \in [0, 1]$ ) : 針對每個事件萃取出之特徵估值
* `Score_e` （ $Score(e)$ ) : 各事件之啟發式風險總分

**【Phase 2: 完整矩陣 - 全域設定與判定】**
* `D_cost` （ $D_{cost}$ ) : 當次運算啟用之代價面向全集 (如：時間、金錢、法律等)
* `Kappa_Q2` （ $\kappa_{Q2}$ ) : 直接證據之最低採納門檻
* `Rate_d` （ $Rate(d_{cost})$ ) : 各代價面向之價值轉換匯率表
* `Weight_d` （ $w_{d_{cost}}$ ) : 各代價面向之加權權重
* `Disaster_Multiplier` (災害節點強度參數) : 當次觸發之非線性放大倍率 (若有)
* `Role_Tolerance` (角色承受力加權) : 各角色 $r_i$ 之特殊承受度參數 (若有啟用)
* `Externality_Warning` (降級聲明旗標) : 移除第三方 $G$ 節點時，未考量外部性成本之警告旗標
* `GlobalNet_r` （ $GlobalNet(r_i)$ ) : 各角色跨事件之全域總淨代價
* `E_bb_count` （ $|E_{bb}|$ ) : Q2 初始評估母體事件總數 (用於對齊後續流程之衰減追蹤)
* `Ratio_unk` （ $Ratio_{unk}$ ) : 全域 UNKNOWN 事件佔比實際值
* `Rho_unk` （ $\rho_{unk}$ ) : 佈署方設定之 UNKNOWN 容忍上限門檻
* `Kappa_global_pass` （ $\kappa_{global\_pass}$ ) : 全域放行門檻
* `Kappa_global_fail` （ $\kappa_{global\_fail}$ ) : 全域攔截門檻
* `Route_Trigger_Q2` (路由觸發紀錄) : 決策落入全域三態邏輯閘之關卡 (FAIL 優先 / 信度否決 / 放行 / 灰階緩衝)
* `Status_Q2_z` （ $Status_{Q2}(z)$ ) : Q2 模組全域最終輸出結果 (PASS / FAIL / UNKNOWN)

**【Phase 2: 完整矩陣 - 單一事件與角色紀錄】**
* `Cov_Q2_e` （ $Cov_{Q2}(e, d_{cost})$ ) : 直接證據完備度估值
* `Hist_Q2_e` （ $Hist_{Q2}(e, d_{cost})$ ) : 歷史常理布林值
* `Status_Q2_e` （ $Status_{Q2}(e, d_{cost})$ ) : 該代價面向之三段式判定狀態 (PASS / FAIL / UNKNOWN)
* `Pay_Receive_raw` （ $Pay_{raw}$ / $Receive_{raw}$ ) : 具備直接證據之原始數值
* `Pay_Receive_est` （ $Pay_{est}$ / $Receive_{est}$ ) : UNKNOWN 狀態下採納之歷史推算值，或記錄為強制歸零
* `Evidence_Q2` （ $Evidence_{Q2}(e, d_{cost})$ ) : 支持該代價採納之直接證據或歷史案例連結
* `Net_raw` （ $Net(e, r_i, d_{cost})$ ) : 各面向原始淨流向矩陣
* `Net_intensity` （ $Net'\_{intensity}(e, r_i, d_{cost})$ ) : 匯率轉換後之強度值矩陣
* `NetTotal_e` （ $NetTotal(e, r_i)$ ) : 單一事件單一角色之總淨代價

### 4. Q3 審計日誌欄位 (Asymmetric Privilege Audit Log)
**【跨模組載入與遮罩運算】**
* `Status_Q1_Load` （ $Status_{Q1}(e)$ ) : 所有輸入事件 $e \in E_{bb}$ 繼承自 Q1 之最終判定狀態快照
* `NetTotal_Load` （ $NetTotal(e, r_i)$ ) : 繼承自 Q2 之各角色原始事件淨代價快照
* `M_deduct` （ $M_{deduct}(e)$ ) : 遮罩陣列 (觸發合法抵銷為 1，保留原價為 0)
* `ResidualNet` （ $ResidualNet(e, r_i)$ ) : 各事件、角色經過遮罩運算後之最終殘留偏移量矩陣

**【全域判定紀錄】**
* `GlobalResidual_r` （ $GlobalResidual(r_i)$ ) : 各角色最終用於判定之全域總殘留代價數值
* `E_eval_count` （ $|E_{eval}|$ ) : 排除 Q1 PASS 後之有效評估事件總數
* `Ratio_unk_Q3` （ $Ratio_{unk}$ ) : 最終算定之 UNKNOWN 比例
* `Thresholds_Q3` （ $\rho_{unk}$, $\kappa_{global\_pass}$, $\kappa_{global\_fail}$ ) : 當次生效之信度與風險門檻
* `Route_Trigger_Q3` (路由觸發紀錄) : 決策落入全域三態邏輯閘之關卡 (FAIL / UNKNOWN 否決 / PASS / UNKNOWN 緩衝)
* `Status_Q3_z` （ $Status_{Q3}(z)$ ) : Q3 模組全域最終輸出結果 (PASS / FAIL / UNKNOWN)(若由 Q2 短路觸發，須註記 `Triggered by Q2 Fast-Pass`)

### 5. Q4 審計日誌欄位 (Authorization Exemption Audit Log)
**【授權遮罩與證據比對】**
* `Status_consent` （ $Status_{consent}(e)$ ) : 每筆殘留事件之授權狀態判定清單 (PASS / FAIL / UNKNOWN)
* `Exclusion_Trigger_Log` (排除條款觸發日誌) : 因缺乏退出機制或資訊不對稱拒絕授權之規則代碼與文本片段
* `FinalNet` （ $FinalNet(e, r_i)$ ) : 套用遮罩抹除後之最終真實剝削代價矩陣

**【最終全域判定】**
* `GlobalFinal_r` （ $GlobalFinal(r_i)$ ) : 各角色最終全域加總值
* `E_final_count` （ $|E_{final}|$ ) : Q4 最終有效評估樣本數
* `Ratio_unk_Q4` （ $Ratio_{unk}$ ) : 最終算定之 UNKNOWN 比例及生效門檻
* `Status_Q4_z` （ $Status_{Q4}(z)$ ) : 全域最終輸出結果 (若由 Q2或Q3 短路觸發，須註記 `Triggered by Q2/(Q3) Fast-Pass`)

### 6. 版本與完整性 (對照與防竄改)
* `Policy_Rule_Version` (規範與參數版本) : 本次運算所依據之規則 / 匯率表 / 門檻設定等統一版本標記
* `Model_Version` (模型版本) : 本次運算萃取與生成所使用之 LLM 模型版本標記 (若有使用)
* `SnapshotHash` (防竄改快照雜湊值) : 對上述所有快照內容計算之 Hash 值，用於完整性校驗與合規稽核驗證
