# Dual-Model Verification Module
> **💡 Execution Standard & Boundaries:**
> All logical computations and verification tasks within this "Dual-Model Verification and Consistency Comparison" are **executed solely and independently by the open-source audit model (white-box)**. The black-box model ($bb$) does not participate in any actual computation during this phase; it acts strictly as the provider of the data to be verified (snapshot $z_{bb}$). To ensure absolute objectivity and reproducibility during the validation process, the open-source audit model must be forcefully set to $T=0$ (Temperature = 0) and operate on a pure, stateless baseline that is strictly reset after each round of audit data recording.

## I. Purpose

This document aims to define an auditable, replayable, and version-controlled dual-model verification mechanism. Its core operational objectives are as follows:

1. **Black-Box Model (BB-LLM, $bb$)**: Responsible for generating and inferring the following content within the full dialogue context (Dialogue Context):
   * Response Intent
   * Role Traceability Results
   * Event Structure (Event Universe)
   * Disaster Node Identification (based on the shared database baseline)

2. **Open-Source Audit Model (OS-LLM, $os$)**: Operates under restricted conditions (zero historical context retention, mandatory $T=0$ baseline) and is responsible for producing baseline data for consistency comparison with the black-box output, specifically including:
   * Role Traceability Results (*Note: As described in the previous section, the OS role traceability analysis here has already incorporated the "response intent" produced by the black-box model as a core input parameter, thereby integrating the role features at the intent level.*)
   * Event Structure (Event Universe)
   * Disaster Node Identification (based on the shared database baseline)
   * Numerical determination of semantic groups under the "2+4 Rules"
   * Permanent solidification storage of the complete Run-level audit log (Run Log)

### Core Protective Value
By establishing a **Consistency Gate**, the system effectively mitigates security vulnerabilities such as jailbreak bypasses (Jailbreak Bypass), missed risk detection, or event structure degradation in the black-box model. Concurrently, through fixed computations and version-controlled recording on the open-source audit side, it ensures that the overall governance architecture possesses high traceability and dynamic correction capabilities.

---

## II. System Boundaries

### 2.1 System Scope of Handling (Included Targets)
The core operation and algorithmic logic of this module revolve strictly around the following targets:
* Any input case or event narrative $x$ (including dialogue historical context) and the "response intent" of the black-box model.
* **Role Traceability Dimension**: Includes the tracking of the role set (Role Set), role relationships, and potentially affected subjects.
* **Event Set Dimension**: Includes the construction of the event tree (Event Tree) and the full set of event nodes (Event Universe).
* **Disaster Node Dimension**: Comparison, detection, and feature tagging executed based on the known shared database $D$ and disaster classification set $C$.

### 2.2 Explicitly Excluded Out-of-Scope Domains (Not Module Objectives)
Based on the principles of value neutrality and technical positioning, the following matters are explicitly excluded from the implementation objectives of this system:
* Determination of legal liability and attribution of subjective fault (this system only provides structural risk disclosure).
* Final adjudication of real-world physical behavior, or any form of basis for legal enforcement.
* Treating any single model output as the absolute, unique correct answer (the core positioning of this system is solely to provide multi-dimensional role traceability comparisons and objective audit results to serve as a foundation for decision support).

## III. Actors and Terminology

### 3.1 Actors
* **User**: The system operator, responsible for inputting cases $x$, dialogue context, or other requests that trigger events.
* **Deployer**: The system administrator with supreme authority, responsible for adjusting system parameters, configuring computation thresholds, maintaining rule bases, and determining the final risk mitigation and blocking strategies.
* **Black-Box Model (BB-LLM, $bb$)**: Responsible for receiving the complete dialogue context, executing primary inferences, and producing the response intent (Response Intent) and initial event structure.
* **Open-Source Audit Model (OS-LLM, $os$)**: Responsible for executing independent verification, feature extraction, and audit solidification. It is forced to run under a pure baseline with zero context retention and a parameter of $T=0$ (mandatory reset per run).
* **Auditor**: Professionals responsible for evaluating decision trajectories and risk strategies based on the solidified Run-level audit logs (Run Logs) during subsequent governance and PDCA processes.

### 3.2 Terminology
* **Role Traceability**: The computational process of objectively deducing and establishing the role set (Role Set), role interactions, and the scope of affected subjects from input parameters or event narratives (**⚠️ Re-emphasized: This process absolutely does not include the determination of liability establishment or the attribution of subjective fault**).
* **Event Universe**: The complete set of event nodes derived from role interactions (including pre-event, mid-event, post-event, and event trees of potential global impact).
* **Disaster Node**: Extreme high-risk event nodes that completely match the disaster types in the known irreversible event chain database $D$ and have been feature-tagged by the system.
* **Semantic Feature Space & Vector Search**: The feature vector space defined for this architecture. The system projects event features and protective rules (e.g., the 2+4 Rules) into fixed-dimensional numerical vectors (Dense Vectors). By utilizing vector search techniques to calculate distance and similarity, it achieves ultra-fast and highly reproducible numerical comparison determinations.
* **Consistency Gate**: The core defense mechanism. By using the output of the open-source audit model ($os$) as the minimum coverage lower bound (Lower Bound), it strictly verifies the risk coverage completeness and structural consistency of the black-box model ($bb$) output. If the validation fails, it triggers a block or a rollback for recalculation.

## IV. Shared Assets

### 4.1 Event Chain Database $D$ (Shared DB)
* **Known Irreversible Event Chain Database ($D$)**: The core template and rule base of the system, containing known irreversible event chains, disaster node definitions, along with their corresponding trigger conditions and associated role closures.
* **Database Version Number ($D_{Version}$)**: The unique version tag used for system identification and alignment, which **must be forcefully written into the audit log (Run Log) of a single execution** to ensure traceback precision.

### 4.2 Disaster Node Type Set $C$
* **Disaster Type Set ($C$)**: The extreme disaster classification dimensions defined by the system (e.g., suicide, severe injury, irreversible harm, etc.), which can be regarded as the classification mapping result after computing database $D$.
* **Classification Version Number ($C_{Version}$)**: The version tag of the disaster classification definitions, which **must be forcefully written into the audit log**.

> **⚠️ Core Consistency Baseline:**
> In the same execution batch (RunID), both the black-box model ($bb$) and the open-source audit model ($os$) **must be forcefully mounted with and use the exact same version** of database $D$ and disaster set $C$. This is the minimum prerequisite to ensure that both models possess an absolute baseline alignment when conducting "disaster node vocabulary" comparisons.
> If this requirement is unattainable due to deployment environment constraints, the system MAY adopt fallback strategies such as "semantic version compatibility checks" or "backward compatibility". However, the deploying party MUST acknowledge and assume the liability that such alternative solutions may result in unintended boundary shifts and risks of misjudgment.

## VI. Architecture & Data Flow

The data flow of the system during this "Dual-Model Verification and Role Traceability" stage is primarily composed of the following three core modules. Their input and output (I/O) interfaces are defined as follows:

### 1. Black-Box Role Traceability Module (bb Core)
* **Input**: Full context $x$, $Intent_{bb}$ (Response Intent)
* **Output**: $Intent_{bb}$ (Response Intent), $R_{bb}$ (Role Set), $E_{bb}$ (Event Set), $E_{cat_{bb}}$ (Disaster Event), $CAT_{any_{bb}}$ (Disaster Flag) (along with necessary structural descriptions)

### 2. Open-Source Role Traceability Module (OS Gate)
* **Input**: De-identification strategy is optional (pre-masking special data, can be omitted for MVP), but strictly enforces "restricted input" (single-turn input only, meaning no historical context is included), $Intent_{bb}$ (Black-box Response Intent)
* **Output**: $R_{os}$ (Role Set), $E_{os}$ (Event Set), $E_{cat_{os}}$ (Disaster Event), $CAT_{any_{os}}$ (Disaster Flag) (used for comparison)

### 3. Role Traceability Consistency Comparison Module
* **Input**: Output from the Black-Box Role Traceability Module ($bb$ variable group), Output from the Open-Source Role Traceability Module ($os$ variable group)
* **Output**: Role traceability consistency comparison results

## VII. Data Snapshots Outputted from Role Traceability Module to Verification Module (Run Snapshot)

### 7.1 BB-LLM ($bb$) Role Traceability Snapshot (Primary Data Source)
Define the $bb$ snapshot:

$$z_{bb} = \langle RunID_{BB\_{Init}}, Timestamp\_{end_{bb}}, x_{bb}, \tau_{bb}, R_{bb}, E_{bb}, E_{cat_{bb}}, CAT_{any_{bb}}, Intent_{bb}, DVersion_{bb}, CVersion_{bb}, ModelVersion_{bb} \rangle$$

### 7.2 OS-LLM ($os$) Role Traceability Snapshot (For Verification)
Define the $os$ snapshot:

$$z_{os} = \langle RunID_{OS\_{Init}}, Timestamp\_{end_{os}}, x'_{os}(\text{Restricted Input}), \tau_{os}, R_{os}, E_{os}, E_{cat_{os}}, CAT_{any_{os}}, DVersion_{os}, CVersion_{os}, ModelVersion_{os} \rangle$$

Where $x'_{os}$ can be:
* Single-turn input (selected for this MVP)
* Or other restricted forms (custom defined based on requirements)

> **💡 Design Notes:**
> * **Omission of $Intent_{bb}$ in OS Snapshot**: $Intent_{bb}$ is excluded from the $os$ snapshot because both snapshots are concurrently sent to the same verification module for comparison. Since the primary data source ($bb$ snapshot) already fully contains this intent data, redundant data transmission is avoided based on I/O optimization principles.
> * **Independent Timestamp Alignment**: Retaining individual timestamps for both models (carrying over the `Timestamp_end` from the previous phase) is necessary to accurately compare and pinpoint the specific time difference and the exact stage of failure, should asynchronous transmission anomalies or system timeouts occur.
> * **Separate Versioning ($C/D Version$)**: The future architectural vision allows deployers to dynamically add or modify the rule base or disaster definition data, potentially even triggering hot-swaps between single-turn interactions. Therefore, both the black-box and white-box must "separately" record the exact version mounted at the exact moment of their computation. This ensures that if versions mismatch, the verification module can immediately identify the discrepancy and accurately log the modified version information.

## VIII. Consistency Gate

> **⚠️ Pre-requisite Version Validation:**
> If $DVersion_{bb} \neq DVersion_{os}$ or $CVersion_{bb} \neq CVersion_{os}$, then $Gate = FAIL$.
> * **Description**: Theoretically, this scenario should not occur. However, if it does, the system must forcefully re-execute the role traceability process using the latest version of the rule base for validation. If the version mismatch still cannot be resolved due to practical limitations, deployers may substitute this with equivalent countermeasures (e.g., semantic version compatibility checks, backward compatibility handling, etc.).

### 8.1 Gate Result
The boolean output of the gate is defined as:
$$Gate(z_{bb}, z_{os}) \in \lbrace PASS, FAIL \rbrace $$

### 8.2 Core Requirements: Disaster Node Coverage and Tier Level Coverage
**Core Rules (Both must be satisfied):**
1. **Disaster Node Coverage**: $E_{cat_{os}}(z_{os}) \subseteq E_{cat_{bb}}(z_{bb})$ or $CAT_{any_{os}} = 0$ 
   * (Note: Data is derived from the role traceability snapshot $z$)
2. **Tier Level Coverage**: $\tau_{os}(z_{os}) \le \tau_{bb}(z_{bb})$

**Logical Determination and Definition:**
* Define $HazardCoverage(z)$ as follows: If all disaster nodes captured by $os$ exist within the disaster node set of $bb$ (i.e., satisfying $E_{cat_{os}} \subseteq E_{cat_{bb}}$), it is considered fully covered.
  *(Description: If the* $CAT_{any}$ *produced by* $os$ *is 0, meaning no disaster nodes are present, this comparison is bypassed).*
* The Tier level determined by the black-box model $bb$ must be **greater than or equal to** the Tier level of the open-source model $os$ to ensure risk assessments are neither missed nor downgraded.
* **FAIL Trigger Conditions**: If $HazardCoverage(z) = 0$ (meaning **at least one** disaster node identified by $os$ is missing from $bb$), or $\tau_{os}(z) > \tau_{bb}(z)$, the gate determination evaluates to $Gate = FAIL$.
---

### 8.3 FAIL Handling
If the comparison result for Disaster Node Coverage or Tier Level Coverage evaluates to $FAIL$, the system must proceed with the following workflow:
1. If the original assessment of the black-box is already at Tier 2, directly enter the "Termination Procedure".
2. Log the failed comparison result and the current parameters comprehensively into the Audit Log.
3. Force the black-box model $bb$ to re-execute the complete role traceability process at the highest protection level, **Tier 2**.
4. After acquiring the new black-box Tier 2 snapshot, compare it against the $os$ snapshot again: if it passes, proceed to the next workflow; if it fails again, enter the "Termination Procedure".

> **💡 Termination Procedure:**
> i. Execute a "Context Reset" on the black-box model.
> ii. Log the termination event and reset action into the Audit Log.
> iii. The system outputs a standard secure response: "The conditions for endorsement have not been met this time. The system will re-establish the evaluation state in safe mode."

* **Note 1**: The system allows a maximum of "one" upgrade and re-run (to prevent infinite loops).
* **Note 2**: The design rationale behind the forced upgrade to Tier 2 is that a severe cognitive discrepancy has occurred between the black-box and white-box models (resulting in a mismatched Tier level or missed disaster nodes), necessitating cautious scrutiny under the highest standards.
* **Note 3**: If the black-box is already at Tier 2 but still misses disaster nodes, it indicates that the input text may possess high obfuscation or present a risk of Prompt Injection. To uphold the baseline security of the system, the termination procedure is directly executed.
* **Note 4**: The above measures serve as the default mechanism designed to meet the strictest security state. In practice, deployers can adjust the FAIL countermeasures based on their own requirements and risk tolerance.

---

### 8.4 Blackbox Context Reset

**Blackbox State Definition:**
Let the dialogue state held by the black-box model before the $t$-th round of inference be $H_t^{bb}$, where $H_t^{bb}$ represents the "contextual memory / historical dialogue" state available to the black-box model (including conversational content from previous rounds and implicit states).

**Reset Operator:**
Define the "Context Reset" operator as $Reset_{bb}(\cdot)$.
Executing this operator must satisfy:
$$Reset_{bb}(H_t^{bb}) \rightarrow H_{t+1}^{bb}$$
And the newly generated state $H_{t+1}^{bb}$ must **absolutely no longer contain** any historical dialogue content from $H_t^{bb}$ (i.e., it must be treated as a pure, entirely new session state).

* **⚠️ Core Boundary Declaration**: The $Reset$ operator acts **solely** to clear the dialogue inference context of the black-box model ($bb$). It **absolutely does not affect** the continuity or the replayable chain of traceability systems such as the Run Snapshot and the Audit Log.

---

### 8.5 Structural Isomorphism / Similarity Tolerance (Adjustable Parameters)

Mainly checks the event set and role set, formalized as:

$$Sim_R(R_{bb}, R_{os}) \in [0, 1]$$
$$Sim_E(E_{bb}, E_{os}) \in [0, 1]$$
$$\lVert \ |R_{bb}| \ge \theta_{cnt_R} \cdot |R_{os}| \ \rVert \in \lbrace 0, 1 \rbrace $$
$$\lVert \ |E_{bb}| \ge \theta_{cnt_E} \cdot |E_{os}| \ \rVert \in \lbrace 0, 1 \rbrace $$
> *(Comparing the similarity and quantity of the role sets and event sets between* $os$ *and* $bb$ *)*

**Gate Structural Conditions:**
$$StructureOK(z) = \lVert Sim_R(R_{bb}, R_{os}) \ge \theta_{sim_R} \rVert \land \lVert Sim_E(E_{bb}, E_{os}) \ge \theta_{sim_E} \rVert \land \lVert \ |R_{bb}| \ge \theta_{cnt_R} \cdot |R_{os}| \ \rVert \land \lVert \ |E_{bb}| \ge \theta_{cnt_E} \cdot |E_{os}| \ \rVert$$

**Parameter and Validation Descriptions:**
* Where $\theta$ represents user-adjustable parameters (defaults: $\theta_{sim_R}$ and $\theta_{sim_E}$ are $0.75$; $\theta_{cnt_R}$ and $\theta_{cnt_E}$ are $0.6$). $|R|$ and $|E|$ only require calculating the quantity, serving as a coverage density check; $Sim$ serves as a coverage scope check. It does not matter if $bb$ or $os$ is larger, as long as the coverage scope is within the set threshold.
* **※ Empty Set Exemption:** If the $os$ role set is empty, both the quantity and similarity conditions for roles are automatically evaluated as $1$ (PASS); if the $os$ event set is empty, both the quantity and similarity conditions for events are also automatically evaluated as $1$ (PASS), bypassing further computation.
* **※ All "quantity threshold" comparisons strictly apply "ceiling (round up)" logic.** For example: If the quantity of $E_{os}$ = $13$ and $\theta_{cnt_E}$ = $0.8$, then $13 \times 0.8 = 10.4$. Rounded up unconditionally, it becomes $11$. Therefore, the quantity of $E_{bb}$ must be $\ge 11$.
* **※** The $StructureOK$ in this section serves as a "coverage + consistency" check and does not guarantee relational graph isomorphism. Relational consistency and risk convergence are handled by the disaster node coverage and Tier upgrade strategy.
* **※ Rationale for Comparison:** This ensures that the granularity and scope of the analysis have baseline coverage, confirming basic consistency between the black-box and open-source models. The disaster node check, on the other hand, strictly ensures that mandatory nodes to be audited are explicitly included.

---

#### 8.5.1 Similarity Components (Avoiding Text-Only Comparison)

To prevent key features from being diluted when aggregating sets into a single vector, and to account for differences in vocabulary granularity and semantic loss between the black-box and open-source models, the similarity calculation in this system employs "**Bi-directional Coverage Similarity**".

*(Note: This metric is solely used to verify that the black-box model's overall perception of the event structure has not severely deviated. Substantial fatal risk protection is managed by the Disaster Node Coverage ($HazardCoverage$) *.)*

**(A) Semantic Projection Function**
Generated numerical representation via the open-source model ($os$) or an independent Embedding model:
$$\pi_R(\cdot): R \rightarrow \mathbb{R}^{d_R}, \quad \pi_E(\cdot): E \rightarrow \mathbb{R}^{d_E}$$
> *It only requires consistency in using the "same version, same calculation method" here.*

**(B) Bi-directional Coverage Similarity Calculation (Output [0,1])**
* **Calculation Logic:** Separately compute the "**degree to which $os$ is covered by $bb$**" and the "**degree to which $bb$ is covered by $os$**", then take the average of the two as the final similarity.

Taking the role set as an example, first define the uni-directional coverage:
* **Coverage of $os$ towards $bb$ (how similar a match can elements of $os$ find in $bb$):**
  $$Cov_{os \rightarrow bb} = \frac{1}{|R_{os}|} \sum_{r_{os} \in R_{os}} \max_{r_{bb} \in R_{bb}} sim(\pi_R(r_{bb}), \pi_R(r_{os}))$$
* **Coverage of $bb$ towards $os$ (how similar a match can elements of $bb$ find in $os$):**
  $$Cov_{bb \rightarrow os} = \frac{1}{|R_{bb}|} \sum_{r_{bb} \in R_{bb}} \max_{r_{os} \in R_{os}} sim(\pi_R(r_{os}), \pi_R(r_{bb}))$$

**Definition of Final Role Set and Event Set Similarity:**
$$Sim_R(R_{bb}, R_{os}) := \frac{Cov_{os \rightarrow bb} + Cov_{bb \rightarrow os}}{2} \in [0, 1]$$
$$Sim_E(E_{bb}, E_{os}) := \frac{Cov_{os \rightarrow bb} + Cov_{bb \rightarrow os}}{2} \in [0, 1]$$

---

#### 8.5.2 Same Tier Re-fetch Rules (Re-fetching based on $os$ parameter quantities)

To prevent the scenario where "**bb fails to meet similarity/quantity thresholds due to insufficient extraction**", the following mandatory re-fetch requirements within the same Tier are defined:

1. **Set the re-fetch target quantity for $bb$:**
   $$N_R^{target} := |R_{os}|, \quad N_E^{target} := |E_{os}|$$
2. Log the results into the Audit Log.
3. Return to the role traceability phase for $bb$ , re-input the parameters, **and explicitly require** $bb$ **to use the specified target quantity** ( $os$ quantity) **as a lower-bound reference**, examining the data with higher sensitivity and re-extracting.
   *(Note: It is not strictly mandated that the absolute quantity produced by* $bb$ *exactly equals* $N^{target}$ *, to prevent the model from hallucinating/fabricating data just to make up the numbers. However, for coverage,* $bb$ *can be required to inspect with a more sensitive/granular state.)*
4. After extraction is complete, the snapshot comparison process must also be re-run. If it fails a second time (identified as the second time if $x_{bb}$ contains $N_R^{target}$ or $N_E^{target}$ parameters), escalate by upgrading the Tier by one level and re-run both the role traceability and snapshot comparison processes, up to Tier 2. (If it is already at Tier 2 after the re-run, directly enter the termination procedure.)
5. **Termination Procedure:**
   i. Execute a Context Reset on the black-box model.
   ii. Record into the Audit Log.
   iii. Output the result: "The conditions for endorsement have not been met this time. The system will re-establish the evaluation state in safe mode."

---

### 8.6 Gate Final Decision

**Gate = PASS must satisfy:**
$$Gate(z_{bb}, z_{os}) = \lVert HazardCoverage(z) = 1 \land StructureOK(z) = 1 \land \tau_{os}(z_{os}) \le \tau_{bb}(z_{bb}) \rVert$$

### 8.7 Audit Log Entry (Result Finalization)

* `AuditRunID`: Unique identifier for the current event / current Gate Run (UUID)
* `Timestamp_start`: Start timestamp of the current event
* `Timestamp_end`: End timestamp of the current event
* `RunID_BB_Init`: BB initial role traceability snapshot ID
* `RunID_OS_Init`: OS initial role traceability snapshot ID
* `Snapshot Version Mismatch Termination`: 0/1 (or PASS/FAIL)
* `HazardCoverageResult`: 0/1 (or PASS/FAIL)
* `TierCoverageResult`: 0/1 (or PASS/FAIL) (Tier Coverage Result)
* `GateResult`: PASS/FAIL (FAIL merely indicates "the current event did not meet endorsement criteria", it is not a judgment on the event itself)
* `Cov_os_to_bb_R`: Coverage value of os towards bb for roles
* `Cov_bb_to_os_R`: Coverage value of bb towards os for roles
* `Cov_os_to_bb_E`: Coverage value of os towards bb for events
* `Cov_bb_to_os_E`: Coverage value of bb towards os for events
* `Theta_simR`: $\theta_{sim_R}$ (Threshold) or Not evaluated
* `Theta_simE`: $\theta_{sim_E}$ (Threshold) or Not evaluated
* `Theta_cntR`: $\theta_{cnt_R}$ (Quantity comparison threshold) or Not evaluated
* `Theta_cntE`: $\theta_{cnt_E}$ (Quantity comparison threshold) or Not evaluated
* `CountCheck_R_Result`: Role quantity comparison result (PASS/FAIL) or Not evaluated
* `CountCheck_E_Result`: Event / Consequence quantity comparison result (PASS/FAIL) or Not evaluated
* `StructureOK`: 0/1 (or PASS/FAIL) or Not evaluated
* `CloseReason`: Reason for closure this time (Enum)
    * `PASS`
    * `FAIL_HAZARD_COVERAGE`
    * `FAIL_TIER_COVERAGE`
    * `FAIL_STRUCTURE`
    * `FAIL_VERSION_MISMATCH`
    * `RERUN_REQUESTED` (Requires BB to re-run role traceability, treated as a new event)
    * `SYSTEM_ERROR` (Not a risk rule judgment; indicates the current process was incomplete or necessary data could not be retrieved, thus this GateRun cannot produce an endorsable conclusion.)
* `RerunRequested`: 0/1
* `RerunTarget`: BB_RoleTrace (Fixed value)
* `RerunTierRequested`: 0/1/2 (Which Tier to use for the next round)
* `NextAuditRunID`: Identifier for the next round's new event (If it cannot be generated immediately, mark as Not evaluated, and RerunRequestID must be preserved)
* `RerunRequestID`: Re-run request identifier (Required, used to link the current and the next event)
* `UserFacingOutputCode`: Output code directed at the user (Not full text; e.g., `UserFacingOutputText`)
* `UserFacingOutputText`: Fixed string "The conditions for endorsement have not been met this time. The system will re-establish the evaluation state in safe mode."

#### "Not evaluated" Rules

* **If `CloseReason \in {FAIL_HAZARD_COVERAGE, FAIL_TIER_COVERAGE, FAIL_VERSION_MISMATCH, SYSTEM_ERROR}`**
  $\rightarrow$ All `Sim` / `Structure` / `Count` fields are marked as Not evaluated
* **If `CloseReason = FAIL_STRUCTURE`**
  $\rightarrow$ Only populate fields that were actually computed; the rest are marked as Not evaluated
* **If `CloseReason = RERUN_REQUESTED`**
  $\rightarrow$ The current event is only populated up to the "moment the rerun was requested"; all fields that would only exist post-rerun are strictly marked as Not evaluated
