# 2+4 守則模組

> 參數來自黑箱模型（BB），由開源審計模型（OS）執行計算

本模組隸屬於分流型安全防護層，其主要目的為在不呼叫大語言模型（LLM）進行推論的前提下，透過純數學矩陣與結構化標籤之交叉比對，對輸入之資料快照進行極速的三態判定（PASS / FAIL / UNKNOWN），並完成資料分流與審計固化。

> ※ **全域參數聲明**：本模組之所有門檻參數皆支援靜態手動配置與預設值冷啟動（Cold Start）。待系統上線並累積充足之人工審計標註（Ground Truth）後，可導入數據回測（Backtesting）機制，透過統計模型實現閾值之動態最佳化調校（Dynamic Auto-tuning）。

### 核心設計原則
* **極速過濾**：時間複雜度須控制於 $O(1)$，故允許的狀況下，全數採用預載入矩陣與邏輯閘判定。
* **零信任意圖**：意圖標籤（Intent）僅於特定語境下生效，不具備越級覆寫絕對違禁特徵之特權。
    * *註：系統架構嚴禁實作任何形式之白名單/黑名單特權放行機制。此類機制將破壞「零信任原則」並成為防護破口；若開發端基於運算效能考量而私設特權清單，需自行承擔防護失效之系統性風險。*
* **證據導向**：缺乏足夠特徵覆蓋率之文本，系統拒絕給予信任判定。(除非無任何事件與角色集合)
* **單步決策**：將計分、意圖疊加、證據覆蓋度一次性三態判定，避免流程冗餘。

## 1. 模組輸入（來自角色追溯 Run Snapshot）
令黑箱模型(bb)的角色追溯模組輸出快照(同雙模型驗證資料)：

$$z_{bb} = \langle RunID_{bb\_{Init}}, Timestamp\_{end_{bb}}, x_{bb}, \tau_{bb}, R_{bb}, E_{bb}, E_{cat_{bb}}, CAT_{any_{bb}}, Intent_{bb}, DVersion_{bb}, CVersion_{bb}, ModelVersion_{bb} \rangle$$

其中：
* $x_{bb}$：本輪輸入敘述
* $\tau_{bb}$：Tier（0 / 1 / 2）
* $R_{bb}$：黑箱模型之角色集合
* $E_{bb}$：黑箱模型之事件集合
* $E_{cat_{bb}}$：黑箱模型之被標記之災害事件集合
* $CAT_{any_{bb}}$：黑箱模型之災害標記（旗標）
* $Intent_{bb}$：由前置模組（角色追溯）產出之結構化回應意圖代碼

## 2. 需求參數（Parameters）

### 2.1 語意特徵字典 $\Sigma$
預先定義之 $k$ 維結構化特徵空間（包含絕對違禁特徵與語境相依特徵）。
$$\Sigma = \langle \sigma_1, \sigma_2, ..., \sigma_k \rangle$$
* $k$：語意特徵維度數
* $\sigma_j$：第 $j$ 個語意特徵定義（以短語/條件描述即可）

$\Sigma$ 是字典版本，為開源模型的版本： $\Sigma Version$ 。

*※由於之後* $z_{bb}$ *資料也要與此資料比對，故要求為同一版本，降低誤差。*

*註：所有特徵一視同仁，風險高低由權重與分數門檻決定。*

### 2.2 守則語意權重（Rule Fingerprints）
六條守則對應權重向量： $W_1 ... W_6$ ，其中 $W_i = (W_{i1}, W_{i2}, ..., W_{ik})$ ，且 $W_{ij} \in [0, 1]$。

對應：
* **i=1**：維護種族延續（群體風險）
* **i=2**：保障自我存續（個體風險）
* **i=3**：不得侵犯人權
* **i=4**：非必要不得干預他人選擇（策略空間不當壓縮）
* **i=5**：不得用謊言或扭曲邏輯換取順從（資訊透明度）
* **i=6**：不得剝奪後悔權與修正權（不可逆）

$W_i$ 也是版本化資產，來自開源審計模型(也可預先進行快照鎖定，只要版本相同)：WVersion。如有必要，守則可增加條件或增設各項權重計算，進行更細緻的檢驗。

### 2.3 事件 $\to$ 語意群向量（由 OS 計算，但來源依 BB 內容）
定義事件語意向量：
$$U(z) = (u_1, u_2, ..., u_k) \text{，且 } u_j \in [0, 1]$$
$U$ 為引用 bb 的結構化輸出（ $R_{bb}、E_{bb}、Intent_{bb} \text{ 與 } x_{bb}$ ），OS 僅負責把它轉成固定格式的向量。（ $E_{cat_{bb}}$ 本來就在 $E$ 集合裡，所以不重複列入）

**ExtractorID / ExtractorVersion：抽取器版本（必須記錄）**
* **ExtractorID（抽取器識別碼）**：代表「目前正在使用的是哪一個抽取器」。在實際佈署時，系統可能會針對不同的語境、語言或業務場景，採用不同的特徵提取演算法或模型。這個 ID 用來明確標記該特徵提取元件的身份。
* **ExtractorVersion（抽取器版本號）**：代表「該抽取器的軟體版本或模型權重版本」。只要抽取器的提取邏輯、內部的局部字典或特徵權重有任何更新，此版本號就會隨之變更。

### 2.4 空事件與空特徵處置機制（Empty Event & Feature Handling）

針對上游抽取之「事件集合為空（ $E_{bb} = \emptyset$ ）」，進而導致「特徵向量未被觸發（ $|U(z)|\_1 = 0$ ）」之極端情境（註：實務上角色集合 $R_{bb}$ 可能包含預設之使用者與 AI 實體，故不列入空值強制條件），系統不再進行後續之證據覆蓋度計算，而是依據黑箱模型評級（ $\tau_{bb}$ ）實施 $O(1)$ 的雙向強制分流：

**2.4.1 安全基底放行（Tier 0 物理短路）**
當系統判定 $\tau_{bb} = 0$，且確認無任何風險事件與特徵（ $E_{bb} = \emptyset \land |U(z)|\_1 = 0$ ）時，視為『無可受評量之風險實體』，系統直接賦予 $PASS_i(z) = 1$ 之基底狀態：
$$\text{若 } \tau_{bb} = 0 \land E_{bb} = \emptyset \land |U(z)|\_1 = 0 \implies PASS_i(z) = 1 \quad (\forall i = 1 \dots 6)$$
* **設計意圖**：確保純淨輸入（如日常問候）能以極速短路放行，避免落入灰階，極大化降低常態營運的算力損耗。

**2.4.2 異常繞過攔截（Tier 1/2 防禦性阻斷）**
若黑箱模型判定存在風險（ $\tau_{bb} \ge 1$ ），但事件集合或特徵向量卻呈現空值（ $E_{bb} = \emptyset \lor |U(z)|\_1 = 0$ ），此為系統邏輯悖論。系統將判定為遭遇惡意越獄繞規（Jailbreak Bypass）或抽取器嚴重失效，強制終止判定：
$$\text{若 } \tau_{bb} \ge 1 \land (E\_{bb} = \emptyset \lor |U(z)|\_1 = 0) \implies \text{直接觸發「終止回應，生成報告」}$$
* **設計意圖**：防堵攻擊者利用混淆手法，使模型雖觸發內部安全警報（Tier 1/2），卻成功隱匿實體事件結構或文字特徵以規避檢查。此狀態不予寬容重跑，直接視同架構失效，攔截並固化審計日誌後，按**02.架構失效處理與安全復原機制 (通則)**之失效模式處置。

### 2.5 意圖投影與正當化參數
* **$\pi_i(\cdot)$（意圖投影函式）**：把結構化意圖代碼 $Intent_{bb}$ 投影為 $k$ 維向量 $\pi_i(Intent_{bb}) \in [0, 1]^k$，表示「該宣稱意圖在守則 $i$ 下，所能正當解釋的特徵型態」。基線實作中各守則共用同一投影（ $\pi_i \equiv \pi$）；必要時可為個別守則特化。版本號 $\Pi Version$。
> 解釋：舉例來說，同一個意圖（如「醫療搶救」）在面對不同守則時，理論上能正當化的特徵極限應有所不同。例如在「維護種族延續」守則下，醫療意圖可以豁免「切開身體」的特徵(為了救援的必要性)；但在「不得侵犯人權」守則下，標準可能又會有所差異(知情同意)（即 $\pi_i$ 應各自獨立）。但考量到此為 MVP 版本，為避免初期需維護過於龐大的意圖對照矩陣，故所有守則先共用同一套意圖補償標準（ $\pi_i \equiv \pi$）進行判定。
* **$\rho_{min} \in (0, 1]$（正當化下限）**：意圖係數之硬下限，確保風險永不歸零。
* **$\rho^F$（意圖之 FAIL 門檻）、 $\rho^P$（意圖之 PASS 門檻）**：意圖軸上的兩道切分。
* **$\varepsilon$**：極小正常數，避免除以零。

### 2.6 意圖正當化係數 $\rho_i(z)$
衡量「在實際內容下，宣稱的意圖有多少風險未被正當解釋」。連續、無清單、有下限：
$$\rho_i(z) = \max \left( \rho_{min} , 1 - \frac{\pi_i(Intent_{bb}) \cdot U(z)}{|U(z)|\_1 + \varepsilon} \right)$$
結果範圍： $\rho_i(z) \in [\rho_{min}, 1]$
*(其中* $\cdot$ *為向量內積（對應分量相乘後相加）；* $|U(z)|\_1$ *為* $U(z)$ *所有分量之和。)*

**說明：**
* 宣稱意圖能良好解釋實際內容（例如「客觀敘事」對上敘述性特徵） $\to$ 內積大 $\to \rho_i$ 小 $\to$ 風險被緩解。
* 內容含大量該意圖無法解釋的特徵（例如宣稱「學術」卻出現逐步操作特徵） $\to$ 內積小 $\to \rho_i$ 趨近 $1 \to$ 不予緩解。

這跟白名單有何差異：此處沒有任何「意圖 $\in$ 核准集合」的集合判定，意圖不會直接決定結果；它只透過一個下限為 $\rho_{min} > 0$ 的連續係數參與。光貼標籤而內容對不上，得到的緩解趨近於零。

## 3. 單步三態聯合決策（FAIL $\to$ PASS $\to$ UNKNOWN）

### 3.1 逐條守則分數（加權對齊）
對每條守則 $i$：
$$Score_i(z) = \sum_j (w_{ij} \cdot u_j)$$
衡量「這段內容在守則 $i$ 下的危險程度」。分數高 = 內容看起來危險；分數低 = 內容看起來安全。

把內容分數與意圖融成單一純量：
$$ScoreRho_i(z) = \rho_i(z) \times Score_i(z)$$
*記號說明：* $ScoreRho$ *表示「經意圖係數加權後」的分數，以與原始分數* $Score_i(z)$ *區別。*

### 3.2 門檻定義
每條守則 $i$ 定義兩個門檻：
* $\theta F_i$：FAIL 門檻
* $\theta P_i$：PASS 門檻
*(且* $0 < \rho_{min} \le \rho^P < \rho^F \le 1$， $\rho_{min} \le \rho^P$ *是為了讓 PASS 可達；否則* $\rho_i$ *永遠大於* $\rho^P$ *，無一可放行。)*

### 3.3 先定義（條件代號化）
$$FailScore_i(z) = \| ScoreRho_i(z) \ge \theta F_i \| \quad \text{(對應輸入：} U(z) \text{ + 意圖加權)}$$
$$PassScore_i(z) = \| ScoreRho_i(z) \le \theta P_i \| \quad \text{(對應輸入：} U(z) \text{ + 意圖加權)}$$
$$FailIntention_i(z) = \| \rho_i(z) \ge \rho^F \| \quad \text{(對應輸入：} Intent_{bb} \text{) (偏惡意)}$$
$$PassIntention_i(z) = \| \rho_i(z) \le \rho^P \| \quad \text{(對應輸入：} Intent_{bb} \text{) (非惡意)}$$

### 3.4 再定義證據充分度
$$Cov(z) \in [0, 1]$$
* $\kappa_i$：第 $i$ 條的最低充分度(是否有充足證據證明，可自行設定)
$$Enough_i(z) = \| Cov(z) \ge \kappa_i \|$$

### 3.5 三態輸出
$$FAIL_i(z) = FailScore_i(z) \cdot Enough_i(z) \cdot FailIntention_i(z)$$
$$PASS_i(z) = PassScore_i(z) \cdot Enough_i(z) \cdot PassIntention_i(z)$$
$$UNKNOWN_i(z) = 1 - \max(FAIL_i(z), PASS_i(z))$$

**Status_i(z) 決策判定：**
* **FAIL** 若 $FAIL_i(z) = 1$
* **PASS** 若 $PASS_i(z) = 1$
* **UNKNOWN** 其餘都是 unknown（即 $UNKNOWN_i(z) = 1$）

## 4. 六條守則判斷處理規則

### 4.1 終止回應，生成報告
$$(Status_1(z) = \text{Fail} \lor Status_2(z) = \text{Fail}) \lor ((Status_1(z) = \text{Unknown} \lor Status_2(z) = \text{Unknown}) \land \tau_{bb} = 2)$$

**步驟：**
1. 將結果計入審計日誌
2. 依審計日誌項目結果生成風險揭露報告
3. 嚴格禁止回應
4. 重置OS

風險揭露報告：列出判定為 unknown、Fail 之規則與其相關驗證紀錄及 $E_{cat}$項目進行生成，可依照需求進行設計，用違反項目回應或者針對情況回應外部者，不對外部者揭露風險報告。例如：「此資訊與傷害有關，故無法生成回應。」

### 4.2 風險非 2 級，黑箱模型 bb 重新以 Tier 2 進行完整角色追溯流程(需先將記錄寫入審計日誌後，重置OS，再開始Tier 2 角色追溯流程)
$$[(Status_1(z) = \text{Unknown} \lor Status_2(z) = \text{Unknown}) \land \tau_{bb} \neq 2] \lor [(\exists Status_{3..6}(z) = \text{Fail}) \land \tau_{bb} \neq 2]$$

### 4.3 回應輸出 (Output)
$$(\forall Status_{1..6}(z) = \text{Pass}) \land (\tau_{bb} = 0)$$

**步驟：**
1. 將結果計入審計日誌
2. 依審計日誌項目結果生成風險揭露報告(如果有 E_cat 存在的話)
3. 進行回應(以回應意圖 $Intent_{bb}$ 為主，進行完整回應生成。可使用較便宜的模型或開源模型進行生成)
4. 重置OS

### 4.4 轉入邊界四問模組
$Otherwise$

### 4.5 結果一覽表(依序判定)

| 風險等級 | 最高規範(1-2) 狀態 | 次要規範(3-6) 狀態 | 最終行動 |
| :--- | :--- | :--- | :--- |
| **任意** | 任一條有 Fail | 任意 | **終止回應，生成報告** |
| **2 級** | 任一條有 Unknown | 任意 | **終止回應，生成報告** |
| **0, 1 級** | 任一條有 Unknown | 任意 | **重新以 Tier 2 進行完整角色追溯** |
| **0, 1 級** | 皆為 Pass | 任一條有 Fail | **重新以 Tier 2 進行完整角色追溯** |
| **0 級** | 皆為 Pass | 皆為 Pass | **直接輸出** |
| **任意** | 皆為 Pass | 任意 | **轉入邊界四問模組** |

## 5. 計入審計日誌（結果凝固）

### 5.1 識別與時間
* AuditRunID：本次事件的唯一識別碼 (UUID)
* Timestamp_start：本次事件開始時間戳
* Timestamp_end：本次事件結束時間戳
* RunID_BB_Init: BB 初始角色追溯快照 ID
* ModuleName（例：2plus4_rule_eval）
* ModuleVersion

### 5.2 輸入引用（可回放）
* InputSnapshotRef { 指向本次用的 Run Snapshot：例如 bb 的 $z_{bb}$ 快照 ID / Hash }
* UpstreamRunID { 上游角色追溯/雙模型驗證的 RunID }
* IntentCode { 本次 $Intent_{bb}$ 之結構化代碼；已含於快照，另記以利意圖稽核 }

### 5.3 版本一致性（可重現）
* WVersion { 2+4 規則權重/指紋版本 }
* SigmaVersion { 語意字典/共同座標系版本 }
* IntentProjVersion { 意圖投影 $\pi$ 之版本 $\Pi Version$ }
* ExtractorID
* ExtractorVersion

### 5.4 向量 / 權重之「可驗證」資訊 { 不存本體 }
* k { 語意群維度數 }
* UHash { 事件 $\to$ 語意群向量 U(z) 的雜湊；不存 U 本體 }
* IntentVecHash { 意圖投影向量 $\pi(Intent_{bb})$ 的雜湊；不存本體 }
* RuleWeightHashSet { 6 條規則各自的 hash：WHash_1..WHash_6；或存一個整體 hash 亦可 }

### 5.5 全模組共用門檻與常數 { 記一次即可 }
* Cov { $Cov(z)$ }
* RhoMin { $\rho_{min}$ }
* RhoFail { $\rho^F$ }
* RhoPass { $\rho^P$ }
* Epsilon { $\varepsilon$ }

### 5.6 每條守則逐條結果 { 報告會用到 }
對 i = 1..6 { 用 JSON array 或表格欄位皆可 } 記錄：
* **Score_i**： $Score_i(z)$ 原始加權分數
* **Rho_i**： $\rho_i(z)$ 意圖正當化係數
* **ScoreRho_i**： $ScoreRho_i(z)$ 意圖加權後有效分數
* **ThetaFail_i**： $\theta F_i$ FAIL 門檻
* **ThetaPass_i**： $\theta P_i$ PASS 門檻
* **Kappa_i**： $\kappa_i$ 該守則最低證據門檻
* **Enough_i**： $Enough_i(z)$ 證據條件 { 0/1 }
* **ScoreCondFail_i**： $ScoreRho_i \ge \theta F_i$ 內容 FAIL 條件 { 0/1 }
* **ScoreCondPass_i**： $ScoreRho_i \le \theta P_i$ 內容 PASS 條件 { 0/1 }
* **IntentCondFail_i**： $\rho_i \ge \rho^F$ 意圖 FAIL 條件 { 0/1 }
* **IntentCondPass_i**： $\rho_i \le \rho^P$ 意圖 PASS 條件 { 0/1 }
* **FailVerdict_i**： $FAIL_i(z)$ 三條件合取結果 { 0/1 }
* **PassVerdict_i**： $PASS_i(z)$ 三條件合取結果 { 0/1 }
* **Status_i**： $Status_i(z)$ PASS / FAIL / UNKNOWN

*記錄三個分項條件（內容/證據/意圖）的目的：當 Status 為 UNKNOWN 時，可從分項看出是哪一道條件卡住（證據不足？分數落灰帶？意圖模糊？），供下游仲裁與調參。如有必要，可於審計日誌中簡短紀錄觸發之事件代碼或證據結果，以利稽核追蹤。*

### 5.7 彙總 { 流程分流 + 報告索引 }
* FailSet { 例如 [1, 4, 6] }
* UnknownSet { 例如 [2, 5] }
* HighestSummaryStatus { 最高規範之彙總狀態，僅做摘要/分流用 }

### 5.8 決策輸出 { 對齊流程圖 }
* DecisionPath { 例如：STOP_ON_HIGHEST_FAIL / STOP_ON_HIGHEST_UNKNOWN_T2 / ENTER_BOUNDARY4 / PASS_TO_OUTPUT }
* CloseReason { 原因枚舉：例如 PASS / FAIL_HIGHEST / UNKNOWN_HIGHEST / ... }
* RerunRequested { 0/1；如果流程要求回滾重跑 }
    * RerunRequestID：這次提出重跑的請求單號 { 用來串 }
    * ParentAuditRunID：若本次是某次重跑的結果，就指回上一筆

## 7. 備註：不可逆節點之政策分流（未納入 MVP Gate 規格）
本 MVP 版本中， $E_{cat}$（災害節點/事件命中）僅負責「是否命中」之確定性判定，不額外細分不可逆節點之處置政策。惟在正式產品或高風險場域導入時，建議在對不可逆節點加入政策標註，以區分：

* **BLOCK（禁止放行）**：命中即不得輸出操作性建議；需進入安全模式輸出（例如拒絕提供方法、改為安全替代訊息）。
* **DISCLOSE（放行但需揭露）**：允許輸出，但必須以風險揭露/安全提醒模式呈現（例如戶外活動風險、意外風險等）。

政策標註建議落點為共用資料庫之節點或類別欄位（例如 C/D 資料庫欄位擴充），使其成為可審計之「查表式決策」，避免由模型臨場推測取代。

此路徑將會改為，抓取 $E_{cat}$ 時， $E_{cat}$ 每個事件額外帶上 **BLOCK/ DISCLOSE** 標籤進入流程；在 2+4 與後續邊界四問檢查時，如果事件違反規則，規則將檢視事件標籤，將 **DISCLOSE** 事件視為 Pass。計入審計日誌時，須將標籤一同計入；所有回應輸出前，只要 $CAT_{any} = 1$ (有不可逆事件)，皆須以風險揭露/安全提醒模式呈現。
