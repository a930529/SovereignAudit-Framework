# 雙模型驗證模組 (Dual-Model Verification Module)
> **💡 執行規範與運算邊界 (Execution Standard & Boundaries)：**
> 本「雙模型驗證與一致性比對」之所有邏輯運算與校驗工作，**全數由開源審計模型（白箱）獨立執行**。黑箱模型（ $bb$ ）於此階段不參與任何實質運算，僅作為待驗證資料（快照 $z_{bb}$ ）之提供者。為確保校驗過程之絕對客觀性與可再現性，開源審計模型於執行本模組時，必須強制設定為 $T=0$（Temperature = 0），且每輪審計資料紀錄後重置之純淨無狀態基線。

## 一、 文件目的 (Purpose)

本文件旨在定義一套具備可審計性、可回放性與可版本化管理之雙模型驗證機制，其核心運作目標如下：

1. **黑箱模型（BB-LLM, $bb$ ）**：在完整對話上下文（Dialogue Context）下，負責生成與推論以下內容：
   * 回應意圖（Response Intent）
   * 角色追溯結果（Role Traceability Results）
   * 事件結構（Event Universe）
   * 災害節點辨識（依據共用資料庫基準）
 
2. **開源審計模型（OS-LLM, $os$ ）**：在受限條件（無歷史上下文殘留、強制 $T=0$ 基線）下運行，負責產出與黑箱輸出進行一致性比對之基準資料，具體包含：
   * 角色追溯結果（*註：依前章節所述，此處之 OS 角色追溯分析，已將黑箱模型產出之「回應意圖」納入作為核心輸入參數，整合過意圖層面之角色特徵*）
   * 事件結構（Event Universe）
   * 災害節點辨識（依據共用資料庫基準）
   * 「2+4 守則」之語意群數值化判定
   * 完整 Run 級審計日誌（Run Log）之固化儲存

### 核心防護價值
系統透過設置**一致性閘門（Consistency Gate）**，有效降低黑箱模型遭遇越獄繞規（Jailbreak Bypass）、風險漏抓或事件結構降級之安全漏洞；同時，藉由開源審計端之固定化計算與版本化記錄，確保整體治理架構具備高度的可追溯性與動態修正能力。

## 二、 系統邊界 (System Boundaries)

### 2.1 系統處理範疇（納入對象）
本模組之核心運作與演算邏輯，嚴格圍繞以下對象展開：
* 任意輸入案例或事件敘述 $x$（含對話歷史上下文）與黑箱模型之「回應意圖」。
* **角色追溯層面**：包含角色集合（Role Set）、角色關係及潛在受影響主體之追溯。
* **事件集合層面**：包含事件樹（Event Tree）與事件節點全集（Event Universe）之建構。
* **災害節點層面**：依據已知共用資料庫 $D$ 與災害分類集合 $C$ 執行之比對、檢測與特徵標記。

### 2.2 明確排除範疇（非本模組目標）
基於價值中立與技術定位原則，以下事項明確排除於本系統之建置目標之外：
* 法律責任之成立判定與主觀過失之歸責（本系統僅提供結構性風險揭露）。
* 對真實世界實體行為之最終裁決，或任何形式之法律強制執行基礎。
* 將任何單次之模型輸出視為絕對之唯一正解（本系統之核心定位，僅在於提供多維度之角色追溯比對與客觀稽核結果，以作為決策支援之基礎）。

## 三、 角色與名詞定義 (Actors and Terminology)

### 3.1 系統角色 (Actors)
* **使用者 (User)**：系統操作主體，負責輸入案例 $x$、對話上下文或其他觸發事件之請求。
* **佈署方 (Deployer)**：系統管理者，具備最高權限，負責調整系統參數、配置運算門檻、維護規則庫及決定最終風險處置與阻斷策略。
* **黑箱大模型 (BB-LLM, $bb$ )**：負責承接完整對話上下文，執行主要推論，產出回應意圖（Response Intent）與初始事件結構。
* **開源審計模型 (OS-LLM, $os$ )**：負責執行獨立驗證、特徵萃取與審計固化。強制於無上下文殘留且參數為 $T=0$（每輪強制重置）之純淨基線下運行。
* **審計 / 稽核者 (Auditor)**：負責於後續治理與 PDCA 流程中，基於固化之 Run 級審計日誌（Run Log）評估決策軌跡與風險策略之專業人員。

### 3.2 名詞定義 (Terminology)
* **角色追溯 (Role Traceability)**：從輸入參數或事件敘述中，客觀推導並建立角色集合（Role Set）、角色互動關係與受影響者範圍之運算過程（**⚠️ 再次強調：本過程絕對不包含責任成立判定與主觀過失歸責**）。
* **事件集合 (Event Universe)**：由角色互動推演而生之完整事件節點集合（包含事前、事中、事後及潛在全域影響之事件樹）。
* **災害節點 (Disaster Node)**：與已知不可逆事件鏈資料庫 $D$ 中之災害類型完全相符，並經系統特徵標記之極端高風險事件節點。
* **語意特徵空間與檢索 (Semantic Feature Space & Vector Search)**：為本架構定義之特徵向量空間。系統將事件特徵與防護守則（如 2+4 守則）投影為具備固定維度之數值向量（Dense Vectors），並透過向量檢索技術進行距離與相似度計算，藉此實現極速且具高再現性之數值化比對判定。
* **一致性閘門 (Consistency Gate)**：核心防禦機制。以開源審計模型（ $os$ ）之輸出作為最低覆蓋下界（Lower Bound），藉此嚴格檢核黑箱模型（ $bb$ ）輸出之風險覆蓋完整性與結構一致性，若校驗失敗則觸發阻斷或退回重算。

## 四、 共用資料資產 (Shared Assets)

### 4.1 事件鏈資料庫 $D$ (Shared DB)
* **已知不可逆事件鏈資料庫 (Known Irreversible Event Chain Database, $D$ )**：系統之核心模板與規則庫，內含已知之不可逆事件鏈、災害節點定義及其對應之觸發條件與關聯角色閉包。
* **資料庫版本號 ( $D_{Version}$ )**：供系統識別與對齊之唯一版本標籤，**必須強制寫入單次執行之審計日誌（Run Log）中**以確保回溯精準度。

### 4.2 災害分類集合 $C$ (Disaster Node Type Set)
* **災害類型集合 ( $C$ )**：系統所定義之極端災害分類維度（如自殺、重傷、不可逆傷害等），可視為資料庫 $D$ 運算後之分類映射結果。
* **分類版本號 ( $C_{Version}$ )**：災害分類定義之版本標籤，**必須強制寫入審計日誌中**。

> **⚠️ 核心一致性基準：**
> 黑箱模型（ $bb$ ）與開源審計模型（ $os$ ）於同一執行批次（RunID）中，必須強制使用完全相同版本之資料庫 $D$ 與災害集合 $C$。此乃確保雙模型進行「災害節點語彙」比對時，達成絕對基準對齊之先決條件。
> 若受限於佈署環境而無法達成，系統得改採「語意版本相容性檢查」或「版本向下兼容」等降級方案。惟佈署方必須知悉並承擔，此類替代方案可能導致非預期之邊界偏移與誤判風險。

$$
\begin{aligned}
S = \lbrace PASS, FAIL, UNKNOWN\rbrace \\
\end{aligned}
$$

### 5.2 基本輸入輸出符號
* $x$ ：使用者輸入案例（含上下文）

## 六、 系統總覽與資料流 (Architecture & Data Flow)

本系統於「雙模型驗證與角色追溯」階段之資料流，主要由以下三個核心模組構成，其輸入與輸出（I/O）介面定義如下：

### 1. 黑箱角色追溯模組 (bb Core)
* **輸入 (Input)**：完整上下文 $x$ 、 $Intent_{bb}$ （回應意圖）
* **輸出 (Output)**： $Intent_{bb}$ （回應意圖）、 $R_{bb}$ （角色集合）、 $E_{bb}$ （事件集合）、 $E_{cat_{bb}}$ （災害事件）、 $CAT_{any_{bb}}$ （災害標記）（以及必要的結構描述）

### 2. 開源角色追溯模組 (OS Gate)
* **輸入 (Input)**：去識別化策略可選（將特殊資料進行預先遮蓋，MVP 可先不用），但強制採「限制輸入」（僅單輪輸入，不含上下文之意思）， $Intent_{bb}$ （黑箱回應意圖）
* **輸出 (Output)**： $R_{os}$ （角色集合）、 $E_{os}$ （事件集合）、 $E_{cat_{os}}$ （災害事件）、 $CAT_{any_{os}}$ （災害標記）（用於比對）

### 3. 角色追溯一致性比對模組
* **輸入 (Input)**：黑箱角色追溯模組之輸出（ $bb$ 變數群）、開源角色追溯模組之輸出（ $os$  變數群）
* **輸出 (Output)**：角色追溯一致性比對結果

## 七、 角色追溯模組輸出至驗證模組之資料快照 (Run Snapshot)

### 7.1 BB-LLM ($bb$) 角色追溯快照（主資料來源）
定義 $bb$ 快照：
$$z_{bb} = \langle RunID_{BB\_Init}, Timestamp\_end_{bb}, x_{bb}, \tau_{bb}, R_{bb}, E_{bb}, E_{cat_{bb}}, CAT_{any_{bb}}, Intent_{bb}, DVersion_{bb}, CVersion_{bb}, ModelVersion_{bb} \rangle$$

### 7.2 OS-LLM ($os$) 角色追溯快照（驗證用）
定義 $os$ 快照：

$$z_{os} = \langle RunID_{OS\_Init}, Timestamp\_end_{os}, x'_{os}(\text{限制輸入}), \tau_{os}, R_{os}, E_{os}, E_{cat_{os}}, CAT_{any_{os}}, DVersion_{os}, CVersion_{os}, ModelVersion_{os} \rangle$$

其中 $x'_{os}$ 可為：
* 單輪輸入（本次 MVP 選擇）
* 或其他限制形式（依需求自行定義）

> **💡 欄位設計說明 (Design Notes)：**
> * **$Intent_{bb}$ 省略原則**：未在 $os$ 快照中包含 $Intent_{bb}$，是因為兩個快照最終皆會並發送入同一個驗證模組進行比對；既然主資料源（ $bb$  快照）已完整夾帶該意圖資料，基於 I/O 最佳化原則，不需重複傳送冗餘資料。
> * **時間戳記 ($Timestamp$) 獨立對齊**：保留兩者各自的時間戳記（接續前一階段之 `Timestamp_end`），是為了在發生非同步傳輸異常或系統超時（Timeout）時，能精準比對並確認延遲發生的具體時間差與失效階段。
> * **規則庫版本 ( $C/D Version$ ) 分別紀錄**：未來的架構擴充構想允許佈署方動態增減規則庫或災害定義資料，甚至可能在單輪對話間發生熱更新（Hot-swap）異動。因此，黑白箱必須「分別」紀錄其運算當下所掛載的確切版次，以確保若版本不一致時，驗證模組能立刻識別差異並將異動版本資訊精確紀錄。

## 八、 一致性閘門 (Consistency Gate)

> **⚠️ 前置版本校驗：**
> 若 $DVersion_{bb} \neq DVersion_{os}$ 或是 $CVersion_{bb} \neq CVersion_{os}$，則 $Gate = FAIL$。
> * **說明**：理論上不應出現此情況。但若實際發生，系統應強制以最新版本號之規則庫重新執行角色追溯流程並進行檢核。若實務上受限而依舊無法解決版本不一致之問題，佈署方可採用等效手段進行取代（例如：改為語意版本相容性檢查、版本向下兼容等方式處理）。

### 8.1 閘門輸出 (Gate Result)
閘門之布林輸出定義為：
$$Gate(z_{bb}, z_{os}) \in \lbrace PASS, FAIL\rbrace$$

### 8.2 核心要求：災害節點覆蓋、Tier 等級覆蓋
**核心規則（以下兩條皆須符合）：**
1. **災害節點覆蓋**：  $E_{cat_{os}}(z_{os}) \subseteq E_{cat_{bb}}(z_{bb})$ 或是  $CAT_{any_{os}} = 0$ 
   * (註：資料皆來自角色追溯快照 $z$)
2. **Tier 等級覆蓋**： $\tau_{os}(z_{os}) \le \tau_{bb}(z_{bb})$

**邏輯判定與定義：**
* 定義 $HazardCoverage(z)$ 如下：若所有 $os$ 抓取到的任意災害節點 $e$，皆存在於 $bb$ 的災害節點集合內，（即滿足 $E_{cat_{os}} \subseteq E_{cat_{bb}}$ ），則視為完全覆蓋。
  *(說明：若* $os$ *產出的* $CAT_{any}$ *為 0，代表無災害節點，則不需進行此比對)。*
* 黑箱模型 $bb$ 所判定之 Tier 等級，必須 **大於或等於** 開源模型 $os$ 之 Tier 等級，以確認風險評估未被遺漏或降級。
* **觸發 FAIL 條件**：若 $HazardCoverage(z) = 0$ （即任一 $os$ 判定之災害節點不存在於 $bb$ 內），或者 $\tau_{os}(z) > \tau_{bb}(z)$，則閘門判定 $Gate = FAIL$。

---

### 8.3 FAIL 處理機制 (FAIL Handling)
若災害節點覆蓋、Tier 等級覆蓋之比對結果為 $FAIL$，系統應循以下流程處置：
1. 若黑箱原先之評級已為 Tier 2 等級，則直接進入「終止程序」。
2. 將比對失敗結果與當下參數完整計入審計日誌（Audit Log）。
3. 強制黑箱模型 $bb$ 重新以最高防護等級 **Tier 2** 執行完整的角色追溯流程。
4. 取得新的黑箱 Tier 2 快照後，再次與 $os$ 快照進行比對：若合格則放行至下一流程；若仍不合格，則進入「終止程序」。

> **💡 終止程序 (Termination Procedure)：**
> i. 對黑箱模型執行「上下文重置 (Context Reset)」。
> ii. 將終止事件與重置動作寫入審計日誌。
> iii. 系統輸出標準安全回應：「本次未達可背書條件，系統將以安全模式重新建立評估狀態。」

* **備註 1**：系統最多僅允許「一次」升級重跑（避免無限迴圈）。
* **備註 2**：強制升級為 Tier 2 的設計考量，在於黑箱與白箱的模型判斷產生了嚴重的認知落差（才會導致 Tier 等級不符或災害節點漏抓），故需以最高標準審慎檢視。
* **備註 3**：若黑箱已為 Tier 2 卻仍漏抓災害節點，表示該輸入文本可能具備高度混淆性，或存在提示詞注入（Prompt Injection）之風險。為確保系統安全底線，故直接進入終止程序。
* **備註 4**：以上措施皆是為符合最嚴格安全狀態之預設機制，實務上，佈署方可依需求與風險承受度，自行調整 FAIL 的應對措施。

---

### 8.4 黑箱上下文重置 (Blackbox Context Reset)

**黑箱狀態定義：**
令黑箱模型在第 $t$ 輪推論前所持有的對話狀態為 $H_t^{bb}$，其中 $H_t^{bb}$ 代表黑箱模型可用的「上下文記憶 / 歷史對話」狀態（包含先前輪次之對話內容與隱式狀態）。

**重置運算子 (Reset Operator)：**
定義「上下文重置」運算子為 $Reset_{bb}()$ 。
執行此運算子後必須滿足：
$$Reset_{bb}(H_t^{bb}) \rightarrow H_{t+1}^{bb}$$
且產生的新狀態 $H_{t+1}^{bb}$ **絕對不再包含** $H_t^{bb}$ 中的任何歷史對話內容（即視為一個純淨的全新會話狀態）。

* **⚠️ 核心邊界聲明**： $Reset$ 運算子**僅**作用於清空黑箱模型（ $bb$ ）的對話推論上下文，**絕對不影響** Run Snapshot 與 Audit Log（審計日誌）等溯源系統之連貫性與可回放鏈。

---

### 8.5 結構同構 / 相似度容錯（可調參數）

主要檢查事件集合與角色集合，形式化為：

$$Sim_R(R_{bb}, R_{os}) \in [0, 1]$$
$$Sim_E(E_{bb}, E_{os}) \in [0, 1]$$
$$\lVert \, |R_{bb}| \ge \theta_{cnt_R} \cdot |R_{os}| \, \rVert \in \lbrace 0, 1\rbrace$$
$$\lVert \, |E_{bb}| \ge \theta_{cnt_E} \cdot |E_{os}| \, \rVert \in \lbrace 0, 1\rbrace$$
> *(比較* $os$ *和* $bb$ *的角色集合與事件集合相似度及數量)*

**Gate 結構條件：**
$$StructureOK(z) = \lVert Sim_R(R_{bb}, R_{os}) \ge \theta_{sim_R} \rVert \land \lVert Sim_E(E_{bb}, E_{os}) \ge \theta_{sim_E} \rVert \land \lVert \, |R_{bb}| \ge \theta_{cnt_R} \cdot |R_{os}| \, \rVert \land \lVert \, |E_{bb}| \ge \theta_{cnt_E} \cdot |E_{os}| \, \rVert$$

**參數與檢核說明：**
* 其中 $\theta$ 為使用者可調參數（ $\theta_{sim_R}$、 $\theta_{sim_E}$ 預設 $0.75$， $\theta_{cnt_R}$、 $\theta_{cnt_E}$ 預設 $0.6$）。 $|R|$ 和 $|E|$ 僅需計算數量，作為覆蓋密度檢核； $Sim$ 作為覆蓋範圍檢核， $bb$ 大或 $os$ 大都可以，只要覆蓋範圍在設定值內就可以。
* **※空集合豁免：** 若 $os$ 角色集合為空值，則角色之數量與相似度條件皆自動判定為 $1$ (PASS)；若 $os$ 事件集合為空值，則事件之數量與相似度條件亦自動判定為 $1$ (PASS)，不需進行計算。
* **※所有「數量門檻」比較一律以「無條件進位」判定。** 舉例： $E_{os}$ 數量 = $13$， $\theta_{cnt_E}$ = $0.8$， $13 \times 0.8 = 10.4$，無條件進位成 $11$， $E_{bb}$ 數量必需 $\ge 11$。
* **※**本節之 $StructureOK$ 屬於「覆蓋性＋一致性」檢核，不保證關係圖同構；關係一致性由災害節點覆蓋與 Tier 升級策略做風險收斂。
* **※比對原因：** 為了確保分析的細緻度與範圍有基本的覆蓋性，確認黑白箱有基本的一致性，而災害節點則是為了確認必定要被審核的節點是否有確實列入。

---

#### 8.5.1 相似度分項（避免只比文字）

為避免將集合聚合成單一向量時導致關鍵特徵遭到稀釋，同時考量黑白箱模型間之語彙顆粒度差異與語意損耗，本系統之相似度計算採用「**雙向覆蓋相似度 (Bi-directional Coverage Similarity)**」進行檢核。

*(註：本項指標僅用於確認黑白箱對事件結構之整體認知無嚴重偏離，實質之致命風險防護已由災害節點覆蓋 HazardCoverage 擔當。)*

**(A) 語意投影函數 (Semantic Projection)**
由開源模型 ( $os$ ) 或獨立之 Embedding 模型產生數值表示：
$$\pi_R(\cdot): R \rightarrow \mathbb{R}^{d_R}, \quad \pi_E(\cdot): E \rightarrow \mathbb{R}^{d_E}$$
> *這裡只要「同一版本、同一計算方式」一致即可。*

**(B) 雙向覆蓋相似度計算 (Output [0,1])**
* **計算邏輯：** 分別計算「**$os$ 被 $bb$ 覆蓋的程度**」與「**$bb$ 被 $os$ 覆蓋的程度**」，並取兩者之平均值作為最終相似度。

以角色集合為例，先定義單向覆蓋度：
* **$os$ 向 $bb$ 之覆蓋度（ $os$ 的元素在 $bb$ 中能找到多像的對應）：**
  
  $$Cov_{os \rightarrow bb} = \frac{1}{|R_{os}|} \sum_{r_{os} \in R_{os}} \max_{r_{bb} \in R_{bb}} sim(\pi_R(r_{bb}), \pi_R(r_{os}))$$
  
* **$bb$ 向 $os$ 之覆蓋度（ $bb$ 的元素在 $os$ 中能找到多像的對應）：**
  
  $$Cov_{bb \rightarrow os} = \frac{1}{|R_{bb}|} \sum_{r_{bb} \in R_{bb}} \max_{r_{os} \in R_{os}} sim(\pi_R(r_{os}), \pi_R(r_{bb}))$$

**定義最終角色集合與事件集合相似度：**

$$Sim_R(R_{bb}, R_{os}) := \frac{Cov_{os \rightarrow bb} + Cov_{bb \rightarrow os}}{2} \in [0, 1]$$
$$Sim_E(E_{bb}, E_{os}) := \frac{Cov_{os \rightarrow bb} + Cov_{bb \rightarrow os}}{2} \in [0, 1]$$

---

#### 8.5.2 同 Tier 重抓規則（以 os 參數數量重抓）

為避免「**bb 因抽取不足導致相似度/數量不達標**」，定義同 Tier 的強制重抓要求：

1. **令 $bb$ 重抓目標數量：**
$$N_R^{target} := |R_{os}|, \quad N_E^{target} := |E_{os}|$$
2. 結果計入審計日誌。
3. 返回 $bb$ 的角色追溯階段，重新輸入參數，**並要求 $bb$ 以指定目標數量($os$ 數量)做為參考下限**，以更高敏感度檢視資料並重新抽取。
   *(註：不強制要求* $bb$ *產出的絕對數量必須精準等於* $N^{target}$ *，以防模型為湊數而產生幻覺捏造，覆蓋度則可以要求* $bb$ *以更靈敏/更細緻的狀態檢視)*
4. 抽取完成後，快照比對流程也要重跑，如果第二次還是不合格（ $x_{bb}$ 有 $N_R^{target}$ 或 $N_E^{target}$ 參數則為第二次），Tier 升一級重跑角色追溯與快照比對流程，直到 Tier 2 為止。（如果重跑後已為 Tier 2，直接進入終止程序。）
5. **終止程序：**
   i. 對黑箱模型進行上下文重置。
   ii. 計入進審計日誌。
   iii. 輸出結果：「本次未達可背書條件，系統將以安全模式重新建立評估狀態。」

---

### 8.6 Gate 最終判定

**Gate = PASS 須符合：**
$$Gate(z_{bb}, z_{os}) = \lVert HazardCoverage(z) = 1 \land StructureOK(z) = 1 \land \tau_{os}(z_{os}) \le \tau_{bb}(z_{bb}) \rVert$$

### 8.7 計入審計日誌 (結果凝固)

* `AuditRunID`: 本次事件 / 本次 Gate Run 的唯一識別碼 (UUID)
* `Timestamp_start`: 本次事件開始時間戳
* `Timestamp_end`: 本次事件結束時間戳
* `RunID_BB_Init`: BB 初始角色追溯快照 ID
* `RunID_OS_Init`: OS 初始角色追溯快照 ID
* `快照版次錯誤終止`: 0/1（或 PASS/FAIL）
* `HazardCoverageResult`: 0/1（或 PASS/FAIL）
* `TierCoverageResult`: 0/1（或 PASS/FAIL）（Tier 覆蓋結果）
* `GateResult`: PASS/FAIL（FAIL 僅表示「本次事件未達可背書」，非對事件進行評論）
* `Cov_os_to_bb_R`: os 向 bb 之角色覆蓋度數值
* `Cov_bb_to_os_R`: bb 向 os 之角色覆蓋度數值
* `Cov_os_to_bb_E`: os 向 bb 之事件覆蓋度數值
* `Cov_bb_to_os_E`: bb 向 os 之事件覆蓋度數值
* `Theta_simR`: $\theta_{sim_R}$ (門檻) 或 Not evaluated
* `Theta_simE`: $\theta_{sim_E}$ (門檻) 或 Not evaluated
* `Theta_cntR`: $\theta_{cnt_R}$ (數量比對門檻) 或 Not evaluated
* `Theta_cntE`: $\theta_{cnt_E}$ (數量比對門檻) 或 Not evaluated
* `CountCheck_R_Result`: 角色數量比對結果 (PASS/FAIL) 或 Not evaluated
* `CountCheck_E_Result`: 事件 / 後果數量比對結果 (PASS/FAIL) 或 Not evaluated
* `StructureOK`: 0/1 (或 PASS/FAIL) 或 Not evaluated
* `CloseReason`: 本次為何結案 (枚舉)
    * `PASS`
    * `FAIL_HAZARD_COVERAGE`
    * `FAIL_TIER_COVERAGE`
    * `FAIL_STRUCTURE`
    * `FAIL_VERSION_MISMATCH`
    * `RERUN_REQUESTED`（要求 BB 重跑角色追溯，視為新事件）
    * `SYSTEM_ERROR`（非風險規則判定結果；表示本次流程未完成或無法取得必要資料，因此本次 GateRun 無法產生可背書結論。）
* `RerunRequested`: 0/1
* `RerunTarget`: BB_RoleTrace (固定值)
* `RerunTierRequested`: 0/1/2 (下一輪用哪個 Tier 跑)
* `NextAuditRunID`: 下一輪新事件的識別碼 (若無法當下生成則 Not evaluated，並必須保留 RerunRequestID)
* `RerunRequestID`: 重跑請求識別碼 (必填，用來串接本次與下一次事件)
* `UserFacingOutputCode`: 對使用者輸出代碼 (非全文；例如 `UserFacingOutputText`)
* `UserFacingOutputText`: 固定字串「本次未達可背書條件，系統將以安全模式重新建立評估狀態。」

#### Not evaluated 規則

* **若 `CloseReason \in {FAIL_HAZARD_COVERAGE, FAIL_TIER_COVERAGE, FAIL_VERSION_MISMATCH, SYSTEM_ERROR}`**
  $\rightarrow$ `Sim` / `Structure` / `Count` 欄位全部 Not evaluated
* **若 `CloseReason = FAIL_STRUCTURE`**
  $\rightarrow$ 僅填到實際計算過的欄位，其餘 Not evaluated
* **若 `CloseReason = RERUN_REQUESTED`**
  $\rightarrow$ 本次事件只填到「提出 rerun 的那一刻」為止；所有 rerun 後才會有的欄位一律 Not evaluated
