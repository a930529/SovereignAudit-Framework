# 2+4 Rule Module



> Parameters are sourced from the Black-Box Model (BB), with calculations executed by the Open-Source Audit Model (OS)



This module belongs to the routing-based security protection layer. Its primary purpose is to perform ultra-fast three-state determinations (PASS / FAIL / UNKNOWN) on input data snapshots through purely mathematical matrices and structured label cross-matching, without calling the Large Language Model (LLM) for inference, thereby completing data routing and audit solidification.



> ※ \*\*Global Parameter Declaration\*\*: All threshold parameters in this module support static manual configuration and default value Cold Starts. Once the system is online and has accumulated sufficient artificial audit annotations (Ground Truth), a Backtesting mechanism may be introduced to achieve Dynamic Auto-tuning of thresholds via statistical models.



\### Core Design Principles

\* \*\*Ultra-fast Filtering\*\*: Time complexity must be controlled at $O(1)$; therefore, under allowable conditions, pre-loaded matrices and logic gate determinations should be fully utilized.

\* \*\*Zero-Trust Intent\*\*: Intent labels (Intent) are only effective within specific contexts and do not possess the privilege to bypass and overwrite absolute prohibited features.

&#x20;   \* \*Note: The system architecture strictly prohibits the implementation of any form of whitelist/blacklist privileged pass mechanisms. Such mechanisms will destroy the "Zero-Trust Principle" and become a security breach; if the development side privately sets up privileged lists based on computational efficiency considerations, they must bear the systemic risk of protection failure.\*

\* \*\*Evidence-Oriented\*\*: For text lacking sufficient feature coverage, the system refuses to grant a trust determination (unless there are no events and role sets).

\* \*\*Single-Step Decision\*\*: Integrates scoring, intent overlay, and evidence coverage into a one-time three-state determination to avoid process redundancy.

\## 1. Module Input (From Role Trace Run Snapshot)

Let the role trace module of the black-box model (BB) output a snapshot (identical to the dual-model verification data):



$$z\_{bb} = \\langle RunID\_{bb\\\_Init}, Timestamp\\\_end\_{bb}, x\_{bb}, \\tau\_{bb}, R\_{bb}, E\_{bb}, E\_{cat\_{bb}}, CAT\_{any\_{bb}}, Intent\_{bb}, DVersion\_{bb}, CVersion\_{bb}, ModelVersion\_{bb} \\rangle$$



Where:

\* $x\_{bb}$: The current round of input description

\* $\\tau\_{bb}$: Tier (0 / 1 / 2)

\* $R\_{bb}$: Role set of the BB model

\* $E\_{bb}$: Event set of the BB model

\* $E\_{cat\_{bb}}$: Marked disaster event set of the BB model

\* $CAT\_{any\_{bb}}$: Disaster marker (flag) of the BB model

\* $Intent\_{bb}$: Structured response intent code generated by the preceding module (Role Trace)

\## 2. Required Parameters (Parameters)



\### 2.1 Semantic Feature Dictionary $\\Sigma$

Pre-defined $k$-dimensional structured feature space (including absolute prohibited features and context-dependent features).



$$\\Sigma = \\langle \\sigma\_1, \\sigma\_2, ..., \\sigma\_k \\rangle$$



\* $k$: Number of semantic feature dimensions

\* $\\sigma\_j$: Definition of the $j$-th semantic feature (can be described by phrases/conditions)



$\\Sigma$ is the dictionary version, which is the version of the OS model: $\\Sigma Version$.

\*※Since the $z\_{bb}$ data will be compared with this data later, it is required to be the same version to reduce errors.\*

\*Note: All features are treated equally; the level of risk is determined by weights and score thresholds.\*



\### 2.2 Rule Semantic Weights (Rule Fingerprints)

Six rules corresponding to weight vectors: $W\_1 ... W\_6$, where $W\_i = (W\_{i1}, W\_{i2}, ..., W\_{ik})$, and $W\_{ij} \\in \[0, 1]$.



Corresponding to:

\* \*\*i=1\*\*: Maintain species continuation (Group Risk)

\* \*\*i=2\*\*: Ensure self-preservation (Individual Risk)

\* \*\*i=3\*\*: Do not violate human rights

\* \*\*i=4\*\*: Do not unnecessarily interfere with others' choices (Improper compression of strategic space)

\* \*\*i=5\*\*: Do not use lies or twisted logic to gain compliance (Information Transparency)

\* \*\*i=6\*\*: Do not deprive the right to regret and amend (Irreversibility)



$W\_i$ is also a versioned asset, sourced from the OS audit model (can also be snapshotted and locked in advance, as long as the version is the same): WVersion. If necessary, rules can add conditions or additional weight calculations for more granular inspection.



\### 2.3 Event $\\to$ Semantic Group Vector

(Calculated by OS, but sourced based on BB content)

Define event semantic vector:



$$U(z) = (u\_1, u\_2, ..., u\_k) \\text{, where } u\_j \\in \[0, 1]$$



$U$ is a structured output referencing BB ($R\_{bb}$, $E\_{bb}$, $Intent\_{bb}$, and $x\_{bb}$), OS is only responsible for converting it into a fixed-format vector. ($E\_{cat\_{bb}}$ is already in the $E$ set, so it is not repeatedly included)



\*\*ExtractorID / ExtractorVersion: Extractor Version (must be recorded)\*\*

\* \*\*ExtractorID (Extractor Identification Code)\*\*: Represents "which extractor is currently being used". In actual deployment, the system may adopt different feature extraction algorithms or models for different contexts, languages, or business scenarios. This ID is used to explicitly mark the identity of this feature extraction component.

\* \*\*ExtractorVersion (Extractor Version Number)\*\*: Represents "the software version or model weight version of the extractor". As long as the extraction logic, internal local dictionary, or feature weights of the extractor have any updates, this version number will change accordingly.



\### 2.4 Empty Set Base State (Physical Short-Circuit Pass Mechanism)

When the system determines $\\tau\_{bb} = 0$ and the feature vector $|U(z)|\_1 = 0$, it is considered an \*\*"unassessable risk entity"\*\*. At this time, the system does not apply the evidence coverage calculation and directly assigns the base state of $PASS\_i(z) = 1$:



$$\\text{If } \\tau\_{bb} = 0 \\land |U(z)|\_1 = 0 \\implies PASS\_i(z) = 1 \\quad (\\forall i = 1 \\dots 6)$$



\* \*\*Design Intent\*\*: To ensure that pure inputs (such as daily greetings, risk-free chit-chat) can be short-circuited and passed directly at $O(1)$ speed when no features are triggered, preventing them from erroneously falling into the `UNKNOWN` gray area due to insufficient evidence ($Enough\_i(z) = 0$), thereby maximizing the reduction of computational power loss and system latency in normal operations.



\### 2.5 Intent Projection and Justification Parameters

\* \*\*$\\pi\_i(\\cdot)$ (Intent Projection Function)\*\*: Projects the structured intent code $Intent\_{bb}$ into a $k$-dimensional vector $\\pi\_i(Intent\_{bb}) \\in \[0, 1]^k$, representing "the feature patterns this claimed intent can justifiably explain under rule $i$". In the baseline implementation, all rules share the same projection ($\\pi\_i \\equiv \\pi$); if necessary, it can be specialized for individual rules. Version number $\\Pi Version$.

> Explanation: For example, the same intent (e.g., "medical rescue") theoretically has different limits of justifiable features when facing different rules. For instance, under the "maintain species continuation" rule, the medical intent can exempt the feature of "cutting the body" (due to the necessity of rescue); but under the "do not violate human rights" rule, the standard might differ (involving informed consent) (i.e., $\\pi\_i$ should be independent). However, considering this is an MVP version, to avoid maintaining an overly massive intent mapping matrix initially, all rules temporarily share the same set of intent compensation standards ($\\pi\_i \\equiv \\pi$) for determination.

\* \*\*$\\rho\_{min} \\in (0, 1]$ (Justification Lower Bound)\*\*: The hard lower bound of the intent coefficient, ensuring risk never returns to zero.

\* \*\*$\\rho^F$ (Intent FAIL Threshold), $\\rho^P$ (Intent PASS Threshold)\*\*: Two cut-offs on the intent axis.

\* \*\*$\\varepsilon$\*\*: A very small positive constant to prevent division by zero.



\### 2.6 Intent Justification Coefficient $\\rho\_i(z)$

Measures "under the actual content, how much of the claimed intent's risk cannot be justifiably explained". Continuous, non-list, with a lower bound:



$$\\rho\_i(z) = \\max \\left( \\rho\_{min} , 1 - \\frac{\\pi\_i(Intent\_{bb}) \\cdot U(z)}{|U(z)|\_1 + \\varepsilon} \\right)$$



Result range: $\\rho\_i(z) \\in \[\\rho\_{min}, 1]$

\*(Where $\\cdot$ is the vector dot product (corresponding components multiplied and then added); $|U(z)|\_1$ is the sum of all components of $U(z)$.)\*



\*\*Explanation:\*\*

\* Claimed intent can well explain the actual content (e.g., "objective narrative" vs. descriptive features) $\\to$ Large dot product $\\to \\rho\_i$ is small $\\to$ Risk is mitigated.

\* Content contains a large number of features that the intent cannot explain (e.g., claiming "academic" but procedural operation features appear) $\\to$ Small dot product $\\to \\rho\_i$ approaches $1 \\to$ Risk is not mitigated.



\*\*How is this different from a whitelist?\*\* There is no set determination of "Intent $\\in$ Approved Set" here; the intent does not directly determine the outcome. It only participates via a continuous coefficient with a lower bound $\\rho\_{min} > 0$. Merely attaching a label without matching content results in mitigation approaching zero.

\## 3. Single-Step Three-State Joint Decision (FAIL $\\to$ PASS $\\to$ UNKNOWN)



\### 3.1 Rule-by-Rule Score (Weighted Alignment)

For each rule $i$:



$$Score\_i(z) = \\sum\_j (w\_{ij} \\cdot u\_j)$$



Measures "the danger level of this content under rule $i$". High score = content looks dangerous; low score = content looks safe.



Merge the content score and intent into a single scalar:



$$ScoreRho\_i(z) = \\rho\_i(z) \\times Score\_i(z)$$



\*Notation: $ScoreRho$ represents the score "after intent coefficient weighting", distinguishing it from the original score $Score\_i(z)$.\*



\### 3.2 Threshold Definitions

Each rule $i$ defines two thresholds:

\* $\\theta F\_i$: FAIL threshold

\* $\\theta P\_i$: PASS threshold

\*(And $0 < \\rho\_{min} \\le \\rho^P < \\rho^F \\le 1$, $\\rho\_{min} \\le \\rho^P$ is to make PASS achievable; otherwise $\\rho\_i$ is always greater than $\\rho^P$, and none can pass.)\*



\### 3.3 Initial Definitions (Condition Codification)

$$FailScore\_i(z) = \\| ScoreRho\_i(z) \\ge \\theta F\_i \\| \\quad \\text{(Corresponding input: } U(z) \\text{ + Intent Weighting)}$$

$$PassScore\_i(z) = \\| ScoreRho\_i(z) \\le \\theta P\_i \\| \\quad \\text{(Corresponding input: } U(z) \\text{ + Intent Weighting)}$$

$$FailIntention\_i(z) = \\| \\rho\_i(z) \\ge \\rho^F \\| \\quad \\text{(Corresponding input: } Intent\_{bb} \\text{) (Leaning malicious)}$$

$$PassIntention\_i(z) = \\| \\rho\_i(z) \\le \\rho^P \\| \\quad \\text{(Corresponding input: } Intent\_{bb} \\text{) (Non-malicious)}$$



\### 3.4 Define Evidence Sufficiency

$$Cov(z) \\in \[0, 1]$$

\* $\\kappa\_i$: The minimum sufficiency for the $i$-th rule (whether there is sufficient evidence, configurable)



$$Enough\_i(z) = \\| Cov(z) \\ge \\kappa\_i \\|$$



\### 3.5 Three-State Output

$$FAIL\_i(z) = FailScore\_i(z) \\cdot Enough\_i(z) \\cdot FailIntention\_i(z)$$

$$PASS\_i(z) = PassScore\_i(z) \\cdot Enough\_i(z) \\cdot PassIntention\_i(z)$$

$$UNKNOWN\_i(z) = 1 - \\max(FAIL\_i(z), PASS\_i(z))$$



\*\*Status\_i(z) Decision Determination:\*\*

\* \*\*FAIL\*\* if $FAIL\_i(z) = 1$

\* \*\*PASS\*\* if $PASS\_i(z) = 1$

\* \*\*UNKNOWN\*\* all others are unknown (i.e., $UNKNOWN\_i(z) = 1$)

\## 4. Judgment Processing Rules for the Six Rules



\### 4.1 Terminate Response, Generate Report

$$(Status\_1(z) = \\text{Fail} \\lor Status\_2(z) = \\text{Fail}) \\lor ((Status\_1(z) = \\text{Unknown} \\lor Status\_2(z) = \\text{Unknown}) \\land \\tau\_{bb} = 2)$$



\*\*Steps:\*\*

1\. Record the result into the audit log

2\. Generate a risk disclosure report based on the audit log items

3\. Strictly prohibit response

4\. Reset OS



Risk Disclosure Report: List the rules judged as Unknown or Fail along with their related verification records and $E\_{cat}$ items for generation. It can be designed according to requirements, responding to external parties with violated items or specific situations, without disclosing the risk report to them. Example: "This information is related to harm, therefore a response cannot be generated."



\### 4.2 Risk is not Tier 2, BB model restarts full role trace process as Tier 2

(Must first write the record to the audit log, reset OS, then start the Tier 2 role trace process)



$$\[(Status\_1(z) = \\text{Unknown} \\lor Status\_2(z) = \\text{Unknown}) \\land \\tau\_{bb} \\neq 2] \\lor \[(\\exists Status\_{3..6}(z) = \\text{Fail}) \\land \\tau\_{bb} \\neq 2]$$



\### 4.3 Response Output (Output)

$$(\\forall Status\_{1..6}(z) = \\text{Pass}) \\land (\\tau\_{bb} = 0)$$



\*\*Steps:\*\*

1\. Record the result into the audit log

2\. Generate a risk disclosure report based on the audit log items (if $E\_{cat}$ exists)

3\. Output response (primarily based on the response intent $Intent\_{bb}$, generating a complete response. Cheaper models or open-source models can be used for generation)

4\. Reset OS



\### 4.4 Transfer to Boundary Four-Question Module

$Otherwise$



\### 4.5 Results Summary Table (Evaluated Sequentially)



| Risk Tier | Primary Rules (1-2) Status | Secondary Rules (3-6) Status | Final Action |

| :--- | :--- | :--- | :--- |

| \*\*Any\*\* | Any is Fail | Any | \*\*Terminate response, generate report\*\* |

| \*\*Tier 2\*\* | Any is Unknown | Any | \*\*Terminate response, generate report\*\* |

| \*\*Tier 0, 1\*\* | Any is Unknown | Any | \*\*Restart full role trace as Tier 2\*\* |

| \*\*Tier 0, 1\*\* | All are Pass | Any is Fail | \*\*Restart full role trace as Tier 2\*\* |

| \*\*Tier 0\*\* | All are Pass | All are Pass | \*\*Direct Output\*\* |

| \*\*Any\*\* | All are Pass | Any | \*\*Transfer to Boundary Four-Question Module\*\* |

\## 7. Remarks: Policy Routing for Irreversible Nodes (Not included in MVP Gate specifications)

In this MVP version, $E\_{cat}$ (Disaster Node / Event Hit) is only responsible for the deterministic judgment of "whether hit or not," without further sub-dividing the disposal policy of irreversible nodes. However, when introducing into official products or high-risk fields, it is recommended to add policy annotations to irreversible nodes to distinguish:



\* \*\*BLOCK (Prohibit Pass)\*\*: Hitting means operational suggestions cannot be output; must enter safe mode output (e.g., refusing to provide methods, switching to safe alternative messages).

\* \*\*DISCLOSE (Pass but requires disclosure)\*\*: Allowed to output, but must be presented in a risk disclosure / safety reminder mode (e.g., outdoor activity risks, accident risks, etc.).



The recommended placement for policy annotations is the node or category field in the shared database (e.g., field expansion in the C/D database), making it an auditable "table-lookup decision," avoiding replacement by on-the-spot model inference.



This path will be modified so that when fetching $E\_{cat}$, each $E\_{cat}$ event will additionally carry a \*\*BLOCK / DISCLOSE\*\* tag into the process; during the 2+4 and subsequent Boundary Four-Question checks, if an event violates a rule, the rule will inspect the event tag and treat a \*\*DISCLOSE\*\* event as a Pass. When recorded in the audit log, the tag must be recorded as well; before all responses are output, as long as $CAT\_{any} = 1$ (contains irreversible events), it must be presented in risk disclosure / safety reminder mode.
