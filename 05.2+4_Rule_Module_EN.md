# 2+4 Rule Module
[English](./05.2+4_Rule_Module_EN.md) | [繁體中文](./05.2+4守則模組_ZH-TW.md)

> Parameters are sourced from the Black-Box Model (BB), with calculations executed by the Open-Source Audit Model (OS)

This module belongs to the routing-based security protection layer. Its primary purpose is to perform ultra-fast three-state determinations (PASS / FAIL / UNKNOWN) on input data snapshots through purely mathematical matrices and structured label cross-matching, without calling the Large Language Model (LLM) for inference, thereby completing data routing and audit solidification.

> ※ **Global Parameter Declaration**: All threshold parameters in this module support static manual configuration and default value Cold Starts. Once the system is online and has accumulated sufficient artificial audit annotations (Ground Truth), a Backtesting mechanism may be introduced to achieve Dynamic Auto-tuning of thresholds via statistical models.

### Core Design Principles

* **Ultra-fast Filtering**: Time complexity must be controlled at $O(1)$; therefore, under allowable conditions, pre-loaded matrices and logic gate determinations should be fully utilized.
* **Zero-Trust Intent**: Intent labels (Intent) are only effective within specific contexts and do not possess the privilege to bypass and overwrite absolute prohibited features.
* *Note: The system architecture strictly prohibits the implementation of any form of whitelist/blacklist privileged pass mechanisms. Such mechanisms will destroy the "Zero-Trust Principle" and become a security breach; if the development side privately sets up privileged lists based on computational efficiency considerations, they must bear the systemic risk of protection failure.*
* **Evidence-Oriented**: For text lacking sufficient feature coverage, the system refuses to grant a trust determination (unless there are no events and role sets).
* **Single-Step Decision**: Integrates scoring, intent overlay, and evidence coverage into a one-time three-state determination to avoid process redundancy.

## 1. Module Input (From Role Trace Run Snapshot)

Let the role trace module of the black-box model (BB) output a snapshot (identical to the dual-model verification data):

$$z_{bb} = \langle RunID_{bb\_{Init}}, Timestamp\_{end_{bb}}, x_{bb}, \tau_{bb}, R_{bb}, E_{bb}, E_{cat_{bb}}, CAT_{any_{bb}}, Intent_{bb}, DVersion_{bb}, CVersion_{bb}, ModelVersion_{bb} \rangle$$

Where:
* $x_{bb}$: The current round of input description
* $\tau_{bb}$: Tier (0 / 1 / 2)
* $R_{bb}$: Role set of the BB model
* $E_{bb}$: Event set of the BB model
* $E_{cat_{bb}}$: Marked disaster event set of the BB model
* $CAT_{any_{bb}}$: Disaster marker (flag) of the BB model
* $Intent_{bb}$: Structured response intent code generated by the preceding module (Role Trace)

## 2. Required Parameters (Parameters)

### 2.1 Semantic Feature Dictionary $\Sigma$

Pre-defined $k$-dimensional structured feature space (including absolute prohibited features and context-dependent features).

$$\Sigma = \langle \sigma_1, \sigma_2, ..., \sigma_k \rangle$$

* $k$: Number of semantic feature dimensions
* $\sigma_j$: Definition of the $j$-th semantic feature (can be described by phrases/conditions)

$\Sigma$ is the dictionary version, which is the version of the OS model: $\Sigma Version$.

*※Since the* $z_{bb}$ *data will be compared with this data later, it is required to be the same version to reduce errors.*

*Note: All features are treated equally; the level of risk is determined by weights and score thresholds.*

### 2.2 Rule Semantic Weights (Rule Fingerprints)

Six rules corresponding to weight vectors: $W_1 ... W_6$, where $W_i = (W_{i1}, W_{i2}, ..., W_{ik})$, and $W_{ij} \in [0, 1]$.

Corresponding to:
* **i=1**: Maintain species continuation (Group Risk)
* **i=2**: Ensure self-preservation (Individual Risk)
* **i=3**: Do not violate human rights
* **i=4**: Do not unnecessarily interfere with others' choices (Improper compression of strategic space)
* **i=5**: Do not use lies or twisted logic to gain compliance (Information Transparency)
* **i=6**: Do not deprive the right to regret and amend (Irreversibility)

$W_i$ is also a versioned asset, sourced from the OS audit model (can also be snapshotted and locked in advance, as long as the version is the same): WVersion. If necessary, rules can add conditions or additional weight calculations for more granular inspection.

### 2.3 Event $\to$ Semantic Group Vector

(Calculated by OS, but sourced based on BB content)
Define event semantic vector:

$$U(z) = (u_1, u_2, ..., u_k) \text{, where } u_j \in [0, 1]$$

$U$ is a structured output referencing BB ( $R_{bb}$, $E_{bb}$, $Intent_{bb}$, and $x_{bb}$), OS is only responsible for converting it into a fixed-format vector. ( $E_{cat_{bb}}$ is already in the $E$ set, so it is not repeatedly included)

**ExtractorID / ExtractorVersion: Extractor Version (must be recorded)**
* **ExtractorID (Extractor Identification Code)**: Represents "which extractor is currently being used". In actual deployment, the system may adopt different feature extraction algorithms or models for different contexts, languages, or business scenarios. This ID is used to explicitly mark the identity of this feature extraction component.
* **ExtractorVersion (Extractor Version Number)**: Represents "the software version or model weight version of the extractor". As long as the extraction logic, internal local dictionary, or feature weights of the extractor have any updates, this version number will change accordingly.

### 2.4 Empty Event & Feature Handling Mechanism

For extreme scenarios where the upstream extraction results in an "empty event set ( $E_{bb} = \emptyset$ )", which subsequently leads to "untriggered feature vectors ( $|U(z)|\_1 = 0$ )" 
(Note: In practice, the role set $R_{bb}$ may contain default user and AI entities, so it is excluded from the mandatory empty-value condition), the system will no longer proceed with subsequent evidence coverage calculations. Instead, it will implement an $O(1)$ bidirectional forced routing based on the black-box model's tier rating ( $\tau_{bb}$):

**2.4.1 Safe Baseline Passthrough (Tier 0 Physical Short-Circuit)**
When the system determines that $\tau_{bb} = 0$, and confirms the absence of any risk events and features ($E_{bb} = \emptyset \land |U(z)|\_1 = 0$), it is considered an "Entity with No Evaluable Risk". The system directly assigns a baseline state of $PASS_i(z) = 1$:
$$\text{If } \tau_{bb} = 0 \land E_{bb} = \emptyset \land |U(z)|\_1 = 0 \implies PASS_i(z) = 1 \quad (\forall i = 1 \dots 6)$$
* **Design Intent**: To ensure that pure inputs (e.g., daily greetings) can be passed through via an ultra-fast short-circuit, preventing them from falling into the gray area and maximizing the reduction of computational overhead during normal operations.

**2.4.2 Anomalous Bypass Interception (Tier 1/2 Defensive Blocking)**
If the black-box model determines that a risk exists ( $\tau_{bb} \ge 1$ ), yet the event set or feature vector is empty ( $E_{bb} = \emptyset \lor |U(z)|\_1 = 0$ ), this constitutes a logical paradox within the system. The system will determine that it has encountered a malicious Jailbreak Bypass or a severe extractor failure, and forcefully terminate the evaluation:
$$\text{If } \tau_{bb} \ge 1 \land (E_{bb} = \emptyset \lor |U(z)|\_1 = 0) \implies \text{Directly trigger "Terminate Response, Generate Report"}$$
* **Design Intent**: To prevent attackers from using obfuscation techniques that trigger internal security alerts (Tier 1/2) within the model while successfully hiding the physical event structure or text features to evade inspection. This state will not be granted a lenient rerun; it will be directly treated as an architectural failure. After intercepting and solidifying the audit log, it will be handled according to the failure mode outlined in **02. Architecture Failure Handling and Security Recovery Mechanism (General Rules)**.

### 2.5 Intent Projection and Justification Parameters

* **$\pi_i(\cdot)$ (Intent Projection Function)**: Projects the structured intent code $Intent_{bb}$ into a $k$-dimensional vector $\pi_i(Intent_{bb}) \in [0, 1]^k$, representing "the feature patterns this claimed intent can justifiably explain under rule $i$". In the baseline implementation, all rules share the same projection ( $\pi_i \equiv \pi$); if necessary, it can be specialized for individual rules. Version number $\Pi Version$.

> Explanation: For example, the same intent (e.g., "medical rescue") theoretically has different limits of justifiable features when facing different rules. For instance, under the "maintain species continuation" rule, the medical intent can exempt the feature of "cutting the body" (due to the necessity of rescue); but under the "do not violate human rights" rule, the standard might differ (involving informed consent) (i.e., $\pi_i$ should be independent). However, considering this is an MVP version, to avoid maintaining an overly massive intent mapping matrix initially, all rules temporarily share the same set of intent compensation standards ( $\pi_i \equiv \pi$) for determination.

* **$\rho_{min} \in (0, 1]$ (Justification Lower Bound)**: The hard lower bound of the intent coefficient, ensuring risk never returns to zero.
* **$\rho^F$ (Intent FAIL Threshold), $\rho^P$ (Intent PASS Threshold)**: Two cut-offs on the intent axis.
* **$\varepsilon$**: A very small positive constant to prevent division by zero.

### 2.6 Intent Justification Coefficient $\rho_i(z)$

Measures "under the actual content, how much of the claimed intent's risk cannot be justifiably explained". Continuous, non-list, with a lower bound:

$$\rho_i(z) = \max \left( \rho_{min} , 1 - \frac{\pi_i(Intent_{bb}) \cdot U(z)}{|U(z)|\_1 + \varepsilon} \right)$$

Result range: $\rho_i(z) \in [\rho_{min}, 1]$
*(Where* $\cdot$ *is the vector dot product (corresponding components multiplied and then added);* $|U(z)|\_1$ *is the sum of all components of* $U(z)$ *.)*

**Explanation:**
* Claimed intent can well explain the actual content (e.g., "objective narrative" vs. descriptive features) $\to$ Large dot product $\to \rho_i$ is small $\to$ Risk is mitigated.
* Content contains a large number of features that the intent cannot explain (e.g., claiming "academic" but procedural operation features appear) $\to$ Small dot product $\to \rho_i$ approaches $1 \to$ Risk is not mitigated.

**How is this different from a whitelist?** There is no set determination of "Intent $\in$ Approved Set" here; the intent does not directly determine the outcome. It only participates via a continuous coefficient with a lower bound $\rho_{min} > 0$. Merely attaching a label without matching content results in mitigation approaching zero.

## 3. Single-Step Three-State Joint Decision (FAIL $\to$ PASS $\to$ UNKNOWN)

### 3.1 Rule-by-Rule Score (Weighted Alignment)

For each rule $i$:

$$Score_i(z) = \sum_j (w_{ij} \cdot u_j)$$

Measures "the danger level of this content under rule $i$". High score = content looks dangerous; low score = content looks safe.

Merge the content score and intent into a single scalar:

$$ScoreRho_i(z) = \rho_i(z) \times Score_i(z)$$

*Notation:* $ScoreRho$ *represents the score "after intent coefficient weighting", distinguishing it from the original score* $Score_i(z)$.

### 3.2 Threshold Definitions

Each rule $i$ defines two thresholds:
* $\theta F_i$: FAIL threshold
* $\theta P_i$: PASS threshold

*(And* $0 < \rho_{min} \le \rho^P < \rho^F \le 1$ *,* $\rho_{min} \le \rho^P$ *is to make PASS achievable; otherwise* $\rho_i$ *is always greater than* $\rho^P$ *, and none can pass.)*

### 3.3 Initial Definitions (Condition Codification)

$$FailScore_i(z) = \| ScoreRho_i(z) \ge \theta F_i \| \quad \text{(Corresponding input: } U(z) \text{ + Intent Weighting)}$$
$$PassScore_i(z) = \| ScoreRho_i(z) \le \theta P_i \| \quad \text{(Corresponding input: } U(z) \text{ + Intent Weighting)}$$
$$FailIntention_i(z) = \| \rho_i(z) \ge \rho^F \| \quad \text{(Corresponding input: } Intent_{bb} \text{) (Leaning malicious)}$$
$$PassIntention_i(z) = \| \rho_i(z) \le \rho^P \| \quad \text{(Corresponding input: } Intent_{bb} \text{) (Non-malicious)}$$

### 3.4 Define Evidence Sufficiency

$$Cov(z) \in [0, 1]$$
* $\kappa_i$: The minimum sufficiency for the $i$-th rule (whether there is sufficient evidence, configurable)

$$Enough_i(z) = \| Cov(z) \ge \kappa_i \|$$

### 3.5 Three-State Output

$$FAIL_i(z) = FailScore_i(z) \cdot Enough_i(z) \cdot FailIntention_i(z)$$
$$PASS_i(z) = PassScore_i(z) \cdot Enough_i(z) \cdot PassIntention_i(z)$$
$$UNKNOWN_i(z) = 1 - \max(FAIL_i(z), PASS_i(z))$$

**Status_i(z) Decision Determination:**
* **FAIL** if $FAIL_i(z) = 1$
* **PASS** if $PASS_i(z) = 1$
* **UNKNOWN** all others are unknown (i.e., $UNKNOWN_i(z) = 1$)

## 4. Judgment Processing Rules for the Six Rules

### 4.1 Terminate Response, Generate Report

$$(Status_1(z) = \text{Fail} \lor Status_2(z) = \text{Fail}) \lor ((Status_1(z) = \text{Unknown} \lor Status_2(z) = \text{Unknown}) \land \tau_{bb} = 2)$$

**Steps:**
1. Record the result into the audit log
2. Generate a risk disclosure report based on the audit log items
3. Strictly prohibit response
4. Reset OS

Risk Disclosure Report: List the rules judged as Unknown or Fail along with their related verification records and $E_{cat}$ items for generation. It can be designed according to requirements, responding to external parties with violated items or specific situations, without disclosing the risk report to them. Example: "This information is related to harm, therefore a response cannot be generated."

### 4.2 Risk is not Tier 2, BB model restarts full role trace process as Tier 2

(Must first write the record to the audit log, reset OS, then start the Tier 2 role trace process)

$$[(Status_1(z) = \text{Unknown} \lor Status_2(z) = \text{Unknown}) \land \tau_{bb} \neq 2] \lor [(\exists Status_{3..6}(z) = \text{Fail}) \land \tau_{bb} \neq 2]$$

### 4.3 Response Output (Output)

$$(\forall Status_{1..6}(z) = \text{Pass}) \land (\tau_{bb} = 0)$$

**Steps:**
1. Record the result into the audit log
2. Generate a risk disclosure report based on the audit log items (if $E_{cat}$ exists)
3. Output response (primarily based on the response intent $Intent_{bb}$, generating a complete response. Cheaper models or open-source models can be used for generation)
4. Reset OS

### 4.4 Transfer to Boundary Four-Question Module

$Otherwise$

### 4.5 Results Summary Table (Evaluated Sequentially)

| Risk Tier | Primary Rules (1-2) Status | Secondary Rules (3-6) Status | Final Action |
| :--- | :--- | :--- | :--- |
| **Any** | Any is Fail | Any | **Terminate response, generate report** |
| **Tier 2** | Any is Unknown | Any | **Terminate response, generate report** |
| **Tier 0, 1** | Any is Unknown | Any | **Restart full role trace as Tier 2** |
| **Tier 0, 1** | All are Pass | Any is Fail | **Restart full role trace as Tier 2** |
| **Tier 0** | All are Pass | All are Pass | **Direct Output** |
| **Any** | All are Pass | Any | **Transfer to Boundary Four-Question Module** |

## 7. Remarks: Policy Routing for Irreversible Nodes (Not included in MVP Gate specifications)

In this MVP version, $E_{cat}$ (Disaster Node / Event Hit) is only responsible for the deterministic judgment of "whether hit or not," without further sub-dividing the disposal policy of irreversible nodes. However, when introducing into official products or high-risk fields, it is recommended to add policy annotations to irreversible nodes to distinguish:

* **BLOCK (Prohibit Pass)**: Hitting means operational suggestions cannot be output; must enter safe mode output (e.g., refusing to provide methods, switching to safe alternative messages).
* **DISCLOSE (Pass but requires disclosure)**: Allowed to output, but must be presented in a risk disclosure / safety reminder mode (e.g., outdoor activity risks, accident risks, etc.).

The recommended placement for policy annotations is the node or category field in the shared database (e.g., field expansion in the C/D database), making it an auditable "table-lookup decision," avoiding replacement by on-the-spot model inference.

This path will be modified so that when fetching $E_{cat}$, each $E_{cat}$ event will additionally carry a **BLOCK / DISCLOSE** tag into the process; during the 2+4 and subsequent Boundary Four-Question checks, if an event violates a rule, the rule will inspect the event tag and treat a **DISCLOSE** event as a Pass. When recorded in the audit log, the tag must be recorded as well; before all responses are output, as long as $CAT_{any} = 1$ (contains irreversible events), it must be presented in risk disclosure / safety reminder mode.
