# Role Tracing Module

> **Execution Specification**: Both the Open-Source Auditing Model (OS) and the Black-Box Model (BB) must independently execute this tracing process.

## 0. Basic Sets and Symbol Definitions

* **Input ($x$)**: The foundational data for the tracing computation, including but not limited to:
  * A single sentence, a full paragraph case, or the current message in a multi-turn dialogue (if it is a black-box model, the complete dialogue context must be attached).
  * **Response Intent**: Both the open-source auditing model and the black-box model must incorporate this intent into the analysis as the baseline for role tracing.
  
    > **💡 Glossary: Response Intent**
    > * **Definition**: Refers to the "core semantic goal" or "intended action" targeted by the model's internal decision-making before generating the final literal response (e.g., preparing to give a specific medical prescription, preparing to provide bypass code, or preparing to refuse to answer).
    > * **Why must it be analyzed?**: If only the final literal text output is audited, it is easily bypassed by user jailbreak prompts or word games (such as writing a novel or role-playing). Extracting the "response intent" as an independent input parameter allows the tracing module to perform role and risk determination directly targeting the model's "true motives," ensuring it is not deceived by superficial text and significantly improving system security. Additionally, this intent record facilitates future audit tracing and makes it easier to hand over to a smaller model to reconstruct a safe response.
    > 
    > **💡 Note: Intent Fallback Mechanism**
    > If the black-box model cannot directly produce structured "response intent" data, the system must obtain the raw response text generated by the black-box and hand it over to the open-source auditing model for reverse engineering and specification extraction, thereby reconstructing the structured intent data and recording it.
    > 
    > **⚠️ Audit Baseline**: All reverse engineering histories, temporary states, and final output records within this mechanism must be mandatorily bound to the corresponding `AuditRunID` to ensure the entire process possesses complete and uninterrupted traceability and reproducibility, meeting compliance audit requirements.

* If the parameters are re-fetched after a "return request from Dual-Model Verification or the 2+4 Rules Module," the input **may** additionally include the following constraint conditions:
    * $N_R^{target}$ (Target role constraint set; input if it exists)
    * $N_E^{target}$ (Target event constraint set; input if it exists)
    * $\tau$ (Designated Tier level; **Note: If the input contains a specific $\tau$ value, the system will skip the initial determination and directly enter step four: preliminary tracing**)

    > **💡 Glossary: Target Constraint Sets ($N_R^{target}$ and $N_E^{target}$)**
    > * **Definition**: The specific role or event sets designated for "forced focus" when downstream protection modules determine an abnormal state and request a rerun.
    > * **Why must this mechanism be included? (Focus Pruning and Anti-Drifting)**: Blindly requesting an LLM to elevate the risk tier and rerun the entire dialogue easily leads to wasted computing power and "semantic/role drifting" (where the model, due to randomness, generates new entities unrelated to the controversy). By introducing these constraint conditions, it acts as giving the model a clear "targeted improvement goal," forcing it to lock 100% of its attention onto the disputed entities and behaviors, thereby achieving precise targeting, saving token costs, and ensuring the system's protection logic converges rapidly.

* **Risk Tier ($\tau$)**: $\tau \in \{0, 1, 2\}$
* **Role Set ($R$)**: $R \subseteq U_R$ ($U_R$ is the universal set of all possible roles defined by the system)
* **Event Universal Set ($E$)**: $E \subseteq U_E$ (Event tree / structured event set)
  
  > ※ **Event Relational Structure Declaration**:
  > Let $E = \{e_1, e_2, ..., e_n\}$, where each event $e_i$ must be bound to the role codes participating in that event (pointing to entities in $R$). Its standard structure is a tuple:
  > 
  > $$e_i = \langle Action, ActorID, TargetID \rangle$$
  > 
  > * **Action**: Objective action/event classification (e.g., hitting someone, cutting a body, restricting freedom).
  > * **ActorID**: The role code initiating or executing this action (the subject, corresponding to the ID in $R$, **can be a plural array (multiple roles)**). If there is no clear initiator, set to `Null`.
  > * **TargetID**: The role code receiving this action (the object, corresponding to the ID in $R$, **can be a plural array (multiple roles)**). If the behavior has no specific target, set to `Null`.
  > 
  > **⚠️ Note (Event Aggregation Optimization):**
  > During the inference process, if the same event (Action) has multiple initiators or targets, they should be merged into the same event tuple to save computing power. For example, in a pile-up car accident scenario, both Passenger 1 and Driver 1 suffer abrasions; in this case, they should not be split but merged and output as:
  > $$e_i = \langle \text{"abrasion"}, \text{Null}, [\text{"Passenger1"}, \text{"Driver1"}] \rangle$$
  > 
  > **Example (Engineering Implementation Perspective):**
  > Assuming the role set determined in $R$ is `{ r1: "armed thug", r2: "hostage", r3: "on-duty police officer" }`, the events extracted in $E$ might be:
  > * $e_1 = \langle \text{"restrict freedom"}, [\text{"r1"}], [\text{"r2"}] \rangle \implies$ Thug restricts hostage's freedom
  > * $e_2 = \langle \text{"restrict freedom"}, [\text{"r3"}], [\text{"r1"}] \rangle \implies$ Police officer restricts thug's freedom

* **Disaster Node Type Set ($C$)**:
  * $C = \{\text{suicide, severe homicide injury, death, permanent disability, statelessness, irreversible harm, unlimited long-term care, indefinite liability...}\}$
* **Known Irreversible Event Chain Database ($D$)**:
  * Refers to the system's built-in template library / rule base. Each entry is an "irreversible event chain template" and comes with its associated role chain conditions.
  * **Mapping Relationship**: Each template $d \in D$ can be mapped to one or more disaster types $c \in C$ (classification mapping is defined by the rule base).

## 1. Risk Condition Check (Boolean Feature Vector)

Let the current input be $x$, and define a set of checkers (predicates) for feature extraction:
$$f(x)=(f_{1}(x),f_{2}(x),...,f_{m}(x))\in\{0,1\}^{m}$$
Where each $f_{i}(x)$ corresponds to a "whether..." condition; it is 1 if the condition is met, and 0 otherwise. The current setting is $m = 11$ (tentative, can be added or adjusted according to future needs).

### Risk Check Indicator List

1. **$f_{1}(x)=||[\text{Third-party role appears}]||$**
2. **$f_{2}(x)=||[x\text{ Cross-tripartite destiny}]||$**
   * **Description**: Cross-tripartite destiny includes situations such as escrow, agency, cost allocation, and cost externalization.
3. **$f_{3}(x)=||[\text{Obvious risk signal}]||$**
4. **$f_{4}(x)=||[\text{Decisional intent}]||$**
5. **$f_{5}(x)=||[\text{Operational advice}]||$**
6. **$f_{6}(x)=||[\text{Specific operational requirement}]||$**
7. **$f_{7}(x)=||[\text{Sensitive domain}]||$**
   * **Description**: Sensitive domains include, but are not limited to, medical, legal, financial, and ethical fields.
8. **$f_{8}(x)=||[\text{Systemic pressure}]||$**
   * **Description**: External pressure from institutional, economic, cultural, informational, or power structures that is sufficient to weaken the subject's capacity for free choice, rendering them "substantively unable to refuse."
   * **Display Format**: The "institutional/economic/cultural/informational/power structure" pressure is mapped to a function of intensity score (e.g., falling within $[0,1]$), where $\tau_{SP}$ is the trigger threshold.
   * **Judgment Formula**: $f_{8}(x)=||[sysPressure(x)\ge\tau_{SP}]||$
9. **$f_{9}(x)=||[\text{Role Shift or Role Conflict}]||$**
   * **Evaluation Formula**: $f_{9}(x)=||[RoleShift(x)\vee RoleConflict(x)]||$
   * **Scenario A - Role Shift (RoleShift)**: The same acting subject is required during the dialogue to switch to another role with different permissions, obligations, or responsibility boundaries, leading to an increase in interaction risk.
     * **Manifestation**: The input requests the same subject $a$ to switch to another role set (with different permission/obligation boundaries).
     * **Practical Examples**:
       * *Jailbreak/Bypass Scenario*: The user demands an AI originally acting as "General Customer Service" to forcefully roleplay as an "Unrestricted Developer Mode" or a "Villain in a Novel" to answer dangerous questions.
       * *Privilege Escalation/Compliance Risk Scenario*: The user requests an AI originally providing only "General Health Education Information" to switch roles and act as "the User's Attending Physician" and issue specific medical prescriptions.
     * $RoleShift(x)=||[\exists a,r\ne r^{\prime}:req(a:r\rightarrow r^{\prime}|x)=1]||$
   * **Scenario B - Role Conflict (RoleConflict)**: The same individual assumes two conflicting role obligations simultaneously within the same event, which cannot be fulfilled concurrently, thereby constituting a role conflict.
     * **Manifestation**: The same subject is simultaneously assigned mutually exclusive obligation sets.
     * **Practical Examples**:
       * *Conflict of Interest/Ethical Risk Scenario*: The user asks an AI acting as an "Independent Objective Judge/Auditor" to simultaneously provide "specific advice on how to bypass this audit/exploit loopholes" (acting as both referee and player, where obligations are mutually exclusive).
     * $RoleConflict(x) = || [\exists a: Incompatible (O_{a}(x)) = 1]||$
     * **Note**: $O_{a}(x)$ is the "subject's obligation set" derived from the input, and $Incompatible$ is the mutually exclusive obligation evaluator (defined by the rule base or constraint set).
10. **$f_{10}(x)=||[\text{Any potential to cause harm}]||$**
11. **$f_{11}(x)=||[\text{Any potential to cause an irreversible event}]||$**

> 💡 **Note**: If necessary, additional conditions can be added or weight calculations can be set for each item to conduct a more granular screening.

## 📌 Module Adjudication and Practical Operational Logic

To ensure the system possesses robustness and traceability in high-risk scenarios, the implementation of this module must adhere to the following principles:

### A. Adjudication Unit and Unknown State Handling (Fail-Safe)
* **Indicator Function $||[\cdot]||$**: This represents a monitored and adjudicated conditional entirety. The content enclosed within it is not a mere textual description, but a system adjudication unit. It outputs 1 when the condition is true, and 0 otherwise.
* **Unknown State (Unknown)**: Based on the principle of conservative governance, when the adjudication result is uncertain (unknown), the system will uniformly treat it as **1**, thereby triggering subsequent risk blocking mechanisms or further adjudication.

### B. Operational Architecture and Data Dependency
* **Dynamic Reading of Data Sources**: The pressure threshold ($\tau_{SP}$) in $f_8$ and the mutually exclusive obligation adjudication table ($Incompatible$) in $f_9$ should be dynamically read by the system from the known rule base.
* **Parallel Execution and Mapping Path**: It is recommended to design the 11 risk conditions as parallel checks to enhance performance.

### C. Audit and Compliance Requirements
* The output result of this module (i.e., the Boolean feature vector $f(x)$ containing 11 dimensions) must be written directly into the underlying audit log of the execution batch (RunID).
* The system must not merely record the final judgment result; it must retain the complete feature verification trajectory to satisfy open-source audit standards and ensure full traceability.

## 2. Counter: Number of Conditions Met

$$n(x)=\sum_{i=1}^{m}f_{i}(x)$$

* **Description**: $n(x)$ is the unified calculation entry point for "one item / two or more items" in the document. This step calculates how many risk conditions the input data meets, serving as the basis for Tier routing.

---

## 3. Tier Routing Function

The Tier level depends entirely on the "number of conditions met":

$$
Tier(x) = \begin{cases} 
2, & \text{if } n(x) \ge 2 \\ 
1, & \text{if } n(x) = 1 \\ 
0, & \text{if } n(x) = 0 
\end{cases}
$$

* **Description**: Performs Tier routing based on the counter value. A counter of 2 or more falls under Tier 2.
* **Tier Rule Summary**:

$$
\begin{aligned}
&f(x) \in \lbrace 0, 1 \rbrace^m \\
&n(x) = \sum f_i(x) \\
&Tier(x) = \| [n \ge 2] \| \cdot 2 + \| [n = 1] \| \cdot 1 \text{（ When } n = 0 \text{ it is naturally } 0\text{）}
\end{aligned}
$$

## 4. Perform Initial Traceability First: Irreversible Event Chain Matching (and Potential Escalation)

### 4.1 Irreversible Event Chain Matching Operator

Define the matching function (returns the set of matched irreversible chains):

$$M_D(x) = \{d \in D \mid d \text{ matches } x\}$$

Each irreversible chain $d$ carries a set of "mandatory roles" $Roles(d)\subseteq U_{R}$.

#### 💡 Definition: Irreversible Chain Role Closure ($Roles(d)$)

For any irreversible event chain template $d\in D$, $Roles(d)$ represents the "role closure" (the complete set of required roles) for that specific event chain.

* **Mandatory Inclusion**:
  * $Roles(d)$ **MUST include**: All roles that have appeared anywhere or are structurally required to be present along the event trace path defined by event $d$.
  * Including but not limited to: Initiating roles, Intermediate roles, Indirect roles, Passive roles, and Structurally-required roles.
* **Non-prunable / Non-selective set**:
  * $Roles(d)$ is a non-prunable set: No role may be excluded based on whether they are perceived as innocent, passive, indirect, or non-direct actors.
* **No Liability Attribution**:
  * $Roles(d)$ **DOES NOT represent** responsibility, fault, causal attribution, or legal obligation.
  * It merely indicates the "Required analytical presence for irreversible-risk coverage" in the irreversible risk analysis.
* **Structural Integrity Baseline**:
  * If any $r\in Roles(d)$ is excluded, the analysis **MUST be considered** as having "incomplete irreversible risk coverage" and is invalid.

---

Thus, the initial role set brought out by the irreversible chain is:

$$R_{irr}(x)=\bigcup_{d\in M_{D}(x)}Roles(d)$$

* **Description**: $R_{irr}(x)$ represents the union of all role closures from the matched irreversible event chains triggered by the input $x$.

---

## 4.2 Escalation Rule (Directly to Tier 2 upon matching an irreversible event chain)

$$
\tau_{1}(x) = \begin{cases} 
2, & \text{if } M_{D}(x) \neq \emptyset \\
\tau_{0}(x), & \text{if } M_{D}(x) = \emptyset 
\end{cases}
$$

And pre-emptively include the irreversible chain roles into the role set (regardless of the original Tier):

$$R \leftarrow R_{irr}(x)$$

## 5. Populate Role Set by Tier

Use three types of "Role Extraction/Expansion Operators" (continuing from the previous $R$ set inclusion, prioritizing coverage, with a default of no liability).

### 💡 Core Principles (Applicable to all Tiers)
* The role traceability and event deduction in this section are solely used to establish the **coverage set** required for event analysis, and do not constitute any establishment of liability, attribution of fault, or adjudicative conclusions.
* During the role traceability stage ($\Phi_{0}/\Phi_{1}/\Phi_{2}$), as long as a role has a **factual connection** with the event narrative or event chain, it is preemptively included in the role set. It shall not be excluded regardless of whether it is perceived as innocent, passive, or merely a bystander.
* Any pruning of roles, attribution of liability, or judgment of fault must NOT be performed during the role traceability stage; these can only be handled during the subsequent role interaction analysis and event structure review.
* The Tier differences defined in this section merely represent the expansion of coverage scope and **do not represent differences in the degree of liability**.

### 💡 Engineering Implementation Contract (Role Extraction Contract)
*(This block is a "Design by Contract (DbC)" specification provided for development engineers to define the absolute boundaries and fail-safes for program implementation.)*
* **Input**: $x$
* **Output**: $R \subseteq U_{R}$
* **Guarantees**:
  * $R$ is a superset of all event-related roles under the current Tier.
  * $R$ MUST NOT encode blame, liability, fault, or responsibility.
  * No role may be excluded at this stage due to perceived innocence or passiveness.
* **Prohibition**:
  * Role extraction ($\Phi_{0}/\Phi_{1}/\Phi_{2}$) MUST NOT perform:
    * responsibility attribution
    * fault inference
    * causal blame pruning
* **Note**:
  * Any pruning or exclusion of roles is only permitted after consequence generation ($\Psi$) and role interaction analysis, **never during** $\Phi$.

### 5.1 Tier 0 Minimal Role Traceability (Literal Coverage)
Tier 0 establishes the minimal coverage role set based solely on visible textual signals:
* Explicit literal roles (persons and items)
* Verb objects / implicit objects
* Does not expand hidden institutional chains; does not presume any responsibility

> Considering risk tolerance and resource constraints, deploying parties may consider excluding conversational meta-roles (e.g., User, AI), or even items, from the set to conserve computational resources.

$$\Phi_{0}(x)=R_{literal}(x) \cup R_{obj}(x)$$
* Description: $R_{obj}(x)$ represents the object/implicit object (e.g., directional object roles like "lend it to him", "give it to that person").

### 5.2 Tier 1 Context-Bounded Role Traceability (Who, What, When, Where, Things Coverage)
Tier 1 includes roles related to "persons, events, time, places, and things" to satisfy contextually identifiable coverage:
* Person, event, time, place, thing
* Does not expand outer institutional / cultural / historical layers
* Does not exclude any potential bystanders or passive recipients at this stage (include as long as they are related to the narrative event)

$$\Phi_{1}(x) = R_{person}(x) \cup R_{event}(x) \cup R_{time}(x) \cup R_{place}(x) \cup R_{thing}(x)$$
* Description: $R_{time}(x)$ is solely for "identifiable event sequences in the narrative" and does not make implicit historical inferences.

### 5.3 Tier 2 Complete Role Traceability (Including "Expected but Absent" and Global Domain Coverage)
Building on the contextual coverage of Tier 1, Tier 2 supplements implicit roles that are "expected but absent" based on institutional positions, as well as domain-oriented roles:
* Implicit roles (e.g., family members, employers, government / regulatory authorities, etc., which might need to be included depending on institutional positioning)
* Domain roles (medical / legal / economic / cultural..., acting as system-oriented roles / nodes)
* Still does not constitute liability determination; merely expands the coverage scope

$$\Phi_{2}(x)=\Phi_{1}(x) \cup R_{implicit}(x) \cup R_{domain}(x)$$
* Description: $R_{implicit}(x)$ refers to invisible roles (family, employers, government... etc., to be included based on institutional position).

### 5.4 Role Set Update Rule (Coverage Inclusion)
Based on the role set $R$ obtained in the previous stages, include the role set of the current Tier branch:

$$R \leftarrow R \cup \Phi_{\tau_1(x)}(x)$$
* Note: "Include" here means "include in the review set", and does not imply liability attribution.

## 6. Generate Event Universe via Role Interaction and Perform Disaster Node Detection (Event Layer $\neq$ Liability Layer)

Define the "Tier-based event generation operators". The Event Universe $E$ generated in this section is used for:
* Risk / Event coverage
* Disaster node detection and escalation
* Supporting subsequent convergence (if adopted); does not equate to a determination of liability establishment.

### Tier 0 Minimum Events (Short-Range Causality)
$$\Psi_{0}(x,R)=E_{short}(x,R)$$
*(Short-range causal chain: syntactic inference models directly applying $X \rightarrow Y$)*

### Tier 1 Partial Events (Predictable Timeline)
$$\Psi_{1}(x,R)=E_{predictable}(x,R)$$
*(Only predictable events on the event timeline; does not expand cross-generationally / society-wide)*

### Tier 2 Complete Events (Global Impact Coverage)
Generate first:
$$\Psi_{2}(x,R)=E_{full}(x,R)$$
*(Pre-event / Mid-event / Post-event + Global dimension + Cross-generational / Society-wide)*

$$E \leftarrow \Psi_{\tau_{1}(x)}(x,R)$$

### Disaster Node Detection
Define node classifier (or tagger):

$$
cat(e) = \begin{cases} 
1, & \text{if } type(e) \in C \\
0, & \text{otherwise} 
\end{cases}
$$

Disaster node set (Only for deterministic sets):
$$E_{cat}=\{e\in E \mid cat(e)=1\}$$

* Description: $cat(e)$ is the "final disaster node tagging function". Its recording should occur after role traceability and the Event Universe are fully expanded; in prior stages, only conditional matching is performed, not formal tagging. ($cat$ detection for escalation purposes is allowed to execute early, but the $cat$ tagging in the audit log is only written during the risk matrix phase.)
* ※ $cat(e)$ is solely to tag the existence of uninsurable / irreversible risk events, and does not indicate that subsequent workflows must be blocked.

Any Disaster Node Existence Flag (Indicates whether a disaster node exists):

$$
CAT_{any} = \begin{cases} 
1, & \text{if } \exists e, cat(e)=1 \\ 
0, & \text{otherwise} 
\end{cases}
$$

## 7. If Disaster Nodes are Found in Event Universe: Escalate + Supplement Disaster Node Role Chains + (If Necessary) Re-run Complete Traceability

This rule is as follows:

### 7.1 Escalation Triggered by Disaster Node
$$
\tau_{2}(x) = \begin{cases} 
2, & \text{if } E_{cat} \neq \emptyset \\ 
\tau_{1}(x), & \text{if } E_{cat} = \emptyset 
\end{cases}$$

### 7.2 Inclusion of Disaster Node Role Chains into Role Set
Each disaster node $e$ corresponds to one (or more) disaster role chains $Roles(e)$ (provided by $D$ or the disaster node library):
$$R \leftarrow R \cup \bigcup_{e\in E_{cat}} Roles(e)$$

### 7.3 Re-run Tier 2 Coverage if Necessary (Still No Liability Pre-judgment)
* If $E_{cat} \neq \emptyset$ and $\tau_{1}(x) \neq 2$: Re-run Tier 2 "coverage expansion"
  * $\tau \leftarrow 2$, $R \leftarrow R \cup \Phi_{2}(x)$, $E \leftarrow \Psi_{2}(x,R)$
* If $E_{cat} \neq \emptyset$ and $\tau_{1}(x) = 2$: Do not re-run $E$; only supplement $R$ and tag $E_{cat}$

* Note: The purpose of escalation and re-running here is for "risk/event coverage and complete role presence"; it still is not a liability attribution determination.
* To prevent resource exhaustion, expansion is forcibly stopped when $R_{t+1} \approx R_{t}$, or a maximum recursion depth can be adopted, e.g., $Depth_{max}=2$, to terminate the loop (value can be custom-defined).

## 8. Summary Formula (Data Representation)
$$\tau=\Gamma(x)$$
$$R=Closure_{R}(x,\tau)$$
$$E=Closure_{E}(x,R,\tau)$$

* Where $\Gamma$ is the composite function of "Coarse Triage + Irreversible Chain Hit Escalation + Disaster Node Escalation"; the two $Closure$ components represent the data expansion operators for roles and events, respectively.In implementation, this can be executed as a "Single Pass" or "Iterative until Convergence". For the MVP phase, to conserve computational resources, adopting a Single Pass is sufficient.

## 9. Write to Audit Log (Result Freezing)

The audit objective of this MVP is "result freezing": After each computation is completed, the input and final output results of that computation must be written to the audit log as a snapshot (Run Snapshot) and permanently stored. This snapshot should be readable original content, allowing direct retrieval of the input/output from that time; hash values are strictly for integrity validation, and preserving solely the hash value in place of the snapshot content is prohibited. This MVP does not require saving intermediate calculation steps or internal model reasoning traces in the audit log; if monitoring/observability data (metrics, telemetry, drift, etc.) of the open-source model is integrated in the future, corresponding fields will be supplemented in an expanded version.

### 9.1 Audit Log Fields (Run Snapshot)
* `RunID` (Unique identifier)
* `Timestamp_start` (Execution start timestamp)
* `Timestamp_end` (Execution end timestamp)
* `Input` : $x$ (Current round input message / case snippet, and specified information from dual-model verification)
* `Risk Condition Check` : $f(x)$ (Every single result must be outputted; not required if a specific Tier is designated for this run)
* `Tier_Initial`: $\tau_0$ (The original level triggered by the $f(x)$ feature matrix)
* `Tier` : $\tau\in\{0,1,2\}$
* `Upgrade_Trigger`: (If an upgrade occurs, record the trigger source, e.g., `Irreversible_Chain_Match` or `Disaster_Node_Detected`; otherwise, record as `None`)
* `Role Set (Current Output)` : $R\subseteq U_{R}$
* `Event Set (Current Output)` : $E\subseteq U_{E}$
* `Disaster Node Type Set` : $E_{cat}$
* `Disaster Node Existence Flag` : $CAT_{any}$
* `Black-Box Response Intent (Both white and black boxes must record this field; the black box logs the generated intent, while the white box can directly use the intent code for identification and logging)`
* `Dual-Model Verification ID` : `AuditRunID` (Record if dual-model verification parameters are present)

### 9.2 Version and Integrity (Referencing and Anti-Tampering)
* `Policy/Rule Version`: The version of rules / templates / classification definitions relied upon for this computation (e.g., $D$ or its unified version tag)
* `Model Version` (If a model is used): The tag of the model version used in this computation
* `SnapshotHash`: Calculate the hash value for the snapshot content (9.1 + 9.2 version fields), used for integrity validation and anti-tampering verification

## 10. Open-Source Audit Model Context Reset and Parameter Initialization
* **Description**: Before each run, the open-source audit model must undergo a full context reset and enforce the parameter **T=0 (Temperature = 0)** (the black-box model, being responsible for generation, is not subject to this strict requirement). If necessary, caches or temporary data that might affect the model's accuracy must also be cleared (adjust according to actual architectural needs).
* **💡 Note: Why enforce T=0?**
  In this architecture, "T" stands for the model's generation Temperature. The core task of the audit module is to execute strict logical judgments and risk feature extraction, not creative writing. Setting T=0 eliminates randomness, ensuring that identical inputs always yield absolutely deterministic outputs. This is crucial for the "reproducibility" of audit results and for preventing hallucinations.
